# 12. Files, Workspaces, Search, Settings, and Commands

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Velocity should adopt the native URI/file/workspace/search/configuration/keybinding stack and add project semantics above it. The current virtual filesystem and custom command plumbing are not production substitutes.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| VS-FW-01 — Local file service | The workbench abstracts file reads, writes, watches, metadata, atomic saves, permissions, encodings, conflicts, and provider capabilities behind URI-based services. | Use Code-OSS file services as the single project substrate. The prototype in-memory filesystem remains useful only for demos/tests, not production editing. | Port Velocity services from path strings to URIs and IFileService; add project/worktree providers where needed.<br>Phase: P1 • Priority: P0 • Sources: VEL-03;VS-EXT |
| VS-FW-02 — File Explorer | A hierarchical explorer supports create, rename, move, copy, delete, drag/drop, multi-select, nesting, reveal, refresh, compact folders, exclusions, and decorations. | Use the native Explorer for hands-on control. Overlay reservation, coworker, candidate, review, generated-file, and risk decorations rather than maintaining a separate tree. | Contribute file decorations, context commands, and mission assignment actions to Explorer.<br>Phase: P1 • Priority: P0 • Sources: VS-UI;VEL-13 |
| VS-FW-03 — Open editors and working sets | Users can inspect open files, dirty resources, pinned editors, recently used files, and group-specific working sets. | Expose current coworker working sets separately from human open editors. Following a coworker can reveal its files without stealing the user's tab focus. | Track work-order resources in the orchestration service and project them into optional views.<br>Phase: P3 • Priority: P1 • Sources: VS-UI;VEL-02 |
| VS-FW-04 — Single-folder workspaces | A folder can be opened as a workspace with local settings, tasks, launch configurations, recommendations, and trust state. | Map a Velocity Project to an opened workspace folder plus project metadata. Avoid hiding the underlying repository model. | Add a `.velocity/` manifest and services while preserving standard `.vscode/` conventions.<br>Phase: P1 • Priority: P0 • Sources: VS-SETTINGS;VEL-02 |
| VS-FW-05 — Multi-root workspaces | Multiple folders can be grouped with per-folder settings, tasks, search scopes, SCM repositories, and extension behavior. | Support monorepos, frontend/backend splits, infrastructure repos, design-system repos, and linked services as one Project while letting missions target specific roots. | Use native workspace folders; add semantic root roles and policy inheritance in `.velocity/project.json`.<br>Phase: P3 • Priority: P1 • Sources: VS-UI |
| VS-FW-06 — Workspace files | A `.code-workspace` file stores folders and workspace-level settings and can be shared or versioned. | Generate or adopt workspace files for complex projects, but keep Velocity-specific roles and mission defaults in separate portable manifests. | Add migration/import tooling and avoid mutating user workspace files without approval.<br>Phase: P3 • Priority: P2 • Sources: VS-SETTINGS |
| VS-FW-07 — Hot exit and backups | Unsaved working copies are backed up and restored after restart or crash according to configured behavior. | Preserve human edits safely. Agent work must not depend on unsaved buffers; autonomous changes should be committed or transactionally persisted in worktrees. | Use working-copy backup services for humans and Git/worktree durability for agents.<br>Phase: P2 • Priority: P0 • Sources: VS-EDITOR |
| VS-FW-08 — File watchers | Native or remote watchers efficiently detect changes, coalesce events, respect exclusions, and invalidate models/search/indexes. | Use file events to rebuild project graph, previews, diagnostics, evidence indexes, and design-token views incrementally. | Subscribe through file services; avoid each Velocity service creating its own polling loop.<br>Phase: P2 • Priority: P0 • Sources: VEL-10;VS-EXT |
| VS-FW-09 — Encoding and line endings | Files can be detected, reopened, and saved with selected encodings and CRLF/LF modes; binary files are identified. | Agents must preserve encoding, BOM, permissions, and line endings unless the mission explicitly changes them. Evidence should flag normalization-only diffs. | Use text-file services and file metadata; configure formatter and Git attributes.<br>Phase: P2 • Priority: P1 • Sources: VS-EDITOR |
| VS-FW-10 — Large and binary file handling | The editor and file service protect performance and offer specialized or readonly handling for files that should not be treated as normal text. | Agent context retrieval must exclude or summarize large/binary files and request explicit permission before rewriting assets. | Add content-size/type classifiers and tool limits before model context or patch generation.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| VS-FW-11 — Global text search | Ripgrep-powered search supports regex, case, word, include/exclude globs, open-editor scope, replace, preview, and result navigation. | Use native Search for humans and as the fast lexical fallback for agents. Search results can seed mission scope and impact evidence. | Expose a bounded search API that returns paths, ranges, snippets, and revision identifiers.<br>Phase: P1 • Priority: P0 • Sources: VOSS-01;VS-UI |
| VS-FW-12 — Search editors | Search results can be saved as editable editor documents, rerun, filtered, and reopened. | Use search editors for durable investigations, migration inventories, TODO audits, dependency sweeps, and pre-checkpoint evidence. | Create evidence links to search editor resources and store queries in mission artifacts.<br>Phase: P3 • Priority: P2 • Sources: VS-EXT |
| VS-FW-13 — Replace preview | Workspace replacements can be reviewed per file and per match before application. | Agents should use the same preview discipline for mechanical edits, with a proposed patch, scope count, and rollback point. | Wrap search-replace operations in previewable workspace edits and checkpoints.<br>Phase: P2 • Priority: P1 • Sources: VS-UI |
| VS-FW-14 — Quick file navigation | Recent files, fuzzy file names, line/column suffixes, and navigation history enable fast movement without browsing trees. | Extend quick navigation with candidate/stable variants and coworker working sets while preserving native prefixes and muscle memory. | Contribute Quick Access providers; do not replace native Quick Open.<br>Phase: P2 • Priority: P1 • Sources: VS-UI |
| VS-FW-15 — Timeline view | A resource timeline combines local history, SCM commits, saves, tests, and extension-provided events. | Add mission assignment, reservation, agent patch, verification run, checkpoint, decision, deployment, and rollback entries to a unified artifact history. | Contribute timeline providers backed by the event/evidence store.<br>Phase: P4 • Priority: P1 • Sources: VS-EXT;VEL-02 |
| VS-FW-16 — Local history | File contents are snapshotted on saves and can be compared or restored independently of Git. | Keep local history for human rescue, but agent checkpoints must remain Git-backed and cross-file consistent. | Reuse native local history; surface it as a recovery option, not primary agent versioning.<br>Phase: P2 • Priority: P2 • Sources: VS-EDITOR |
| VS-FW-17 — Settings scopes | Settings resolve across defaults, user, remote, workspace, folder, language, profile, policy, and application scopes. | Define Velocity settings with clear scope: personal ergonomics, project behavior, organization policy, model/provider config, and mission overrides must not collide. | Contribute typed configuration with descriptions, deprecation migrations, and policy eligibility.<br>Phase: P2 • Priority: P0 • Sources: VS-SETTINGS |
| VS-FW-18 — Settings UI and JSON | A graphical editor and JSON editor provide search, filters, modified indicators, scope switching, validation, and direct documentation. | Integrate Velocity settings into native Settings. Use dedicated sheets only for mission/coworker objects that are not ordinary configuration. | Contribute configuration schemas and settings-editor metadata.<br>Phase: P2 • Priority: P1 • Sources: VS-SETTINGS |
| VS-FW-19 — Configuration language overrides | Settings can differ by language or combinations of languages, with precedence rules. | Allow formatters, test commands, agent tools, approval rules, and context budgets to vary by language without custom condition code. | Use language-specific configuration sections and repository defaults.<br>Phase: P3 • Priority: P2 • Sources: VS-SETTINGS |
| VS-FW-20 — Keybinding resolver | Commands are bound through user/default rules, chords, platform mappings, context predicates, conflict display, and removal rules. | Migrate Velocity's existing VS Code-style engine to the native resolver. Ship familiar defaults, but never intercept keys outside command registration. | Register all actions as commands and contribute default keybindings with `when` clauses.<br>Phase: P1 • Priority: P0 • Sources: VEL-03;VS-KEYS |
| VS-FW-21 — Keyboard Shortcuts editor | Users can search commands, inspect conflicts, record keys, edit JSON, sort by precedence, and see source. | Velocity commands must appear here with meaningful categories and titles; hidden one-off shortcuts are prohibited. | Contribute command metadata and default keybindings; add command tests.<br>Phase: P2 • Priority: P1 • Sources: VS-KEYS |
| VS-FW-22 — Command arguments and URIs | Commands can accept structured arguments, be invoked by UI contributions, keybindings, extensions, tests, and command URIs. | Model all important actions as typed commands so humans, agents, automations, deep links, and tests share one execution path. | Create a schema/version layer around Velocity command arguments and authorization.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| VS-FW-23 — Snippets, tasks, and repository recommendations | Workspaces can recommend extensions, define snippets, tasks, debug configurations, and settings that travel with the repository. | A Velocity project should ingest existing developer configuration rather than inventing a parallel ecosystem. Agents must honor repository conventions. | Read `.vscode`, package scripts, devcontainer files, CI workflows, and tool manifests into project context.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS;VS-EXT |
| VS-FW-24 — Workspace Trust | Untrusted folders enter restricted mode, disabling or limiting tasks, debugging, extensions, and other code execution. | Velocity must begin from this trust boundary. Autonomous coworkers cannot execute terminal/browser/MCP actions until the workspace and relevant capabilities are trusted. | Integrate with workspace trust requests and add mission-level permission envelopes above it.<br>Phase: P1 • Priority: P0 • Sources: VS-TRUST;VS-AGENTS |
| VS-FW-25 — Portable mode and data directories | Application data can be isolated to a portable directory for removable, air-gapped, or self-contained deployments. | Offer a portable/local-only Velocity distribution for regulated teams and offline projects, with explicit model/cache/artifact storage locations. | Retain Code-OSS portable mode and make provider downloads, browser profiles, and evidence storage relocatable.<br>Phase: P7 • Priority: P3 • Sources: VS-REMOTE |
| VS-FW-26 — File system providers | Extensions can mount virtual or remote schemes with capability flags for reading, writing, watching, copying, and case sensitivity. | Use providers for remote worktrees, cloud agent artifacts, object storage, generated evidence, and read-only historical snapshots while maintaining URI identity. | Implement only well-bounded providers and ensure tools negotiate capabilities rather than assume local POSIX paths.<br>Phase: P5 • Priority: P1 • Sources: VS-EXT |
