# 10. VS Code Workbench and Interaction Model

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Core workbench behavior should be inherited from Code-OSS. Velocity's work is to register its project concepts through native commands, context keys, editor inputs, views, layout profiles, accessibility services, and restrained product defaults.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| VS-WB-01 — Multi-region workbench | The window composes a primary editor area with an activity bar, primary and secondary side bars, panel, title area, status bar, and notification surfaces. Regions can be hidden, moved, resized, focused, or restored through commands. | Retain Velocity's calm central stage, but implement every Lens, rail, drawer, review sheet, and coworker surface as a registered workbench part or view container rather than a parallel React shell. | Fork-level workbench contributions plus a Velocity layout service that owns default visibility, focus mode, and context-sensitive placement.<br>Phase: P1 • Priority: P0 • Sources: VS-UI;VOSS-01 |
| VS-WB-02 — Editor groups and split layouts | Editors can be arranged into horizontal or vertical groups, dragged between groups, pinned, previewed, locked, moved into new windows, and reopened from history. | Map Velocity panes to native editor groups. Lens layouts become saved group arrangements, with Stable/Candidate comparison and Follow Mode expressed as group orchestration instead of custom split-tree code. | Use IEditorGroupsService, editor inputs, group locking, auxiliary windows, and custom editor serializers.<br>Phase: P1 • Priority: P0 • Sources: VS-UI;VEL-02 |
| VS-WB-03 — Tabs and open-editor management | Tabs expose dirty state, preview state, pinning, close behavior, group membership, and context menus; open editors can be navigated without a permanent tab strip. | Use quiet tabs by default while preserving full keyboard access. Candidate builds, browser sessions, diff reviews, notebooks, and design canvases each become first-class editor inputs. | Implement custom editor input types and serializers; expose commands in Quick Open and the command center.<br>Phase: P1 • Priority: P1 • Sources: VS-UI;VS-EXT |
| VS-WB-04 — Command Palette | Commands from the core and extensions are searchable from one keyboard-first surface, filtered by context and keybinding. | Make every Velocity action—new mission, staff coworker, follow, pause, compare, checkpoint, rollback, ship, reserve scope—an ordinary command. The UI then becomes a view onto the command graph, not the only way to act. | Register commands and menus with context keys; avoid ad-hoc DOM events in the production shell.<br>Phase: P1 • Priority: P0 • Sources: VS-KEYS;VEL-03 |
| VS-WB-05 — Quick Open and symbol navigation | Quick Open switches among files, recent editors, commands, line numbers, workspace symbols, and other providers using prefixes and fuzzy matching. | Extend Quick Open with missions, checkpoints, decisions, coworkers, browser tabs, test runs, deployments, data objects, and semantic project entities. | Create Quick Access providers backed by Velocity's project graph and evidence index.<br>Phase: P2 • Priority: P1 • Sources: VS-UI;VS-EXT |
| VS-WB-06 — Context keys and when clauses | Commands, menus, keybindings, and view visibility are conditionally enabled through a declarative context-key expression system. | Represent project state, active Lens, selected evidence, agent risk, trust state, active reservation, and review status as context keys. This keeps the interface adaptive without hard-coded branching. | Define a namespaced `velocity.*` context-key vocabulary and central state synchronizer.<br>Phase: P1 • Priority: P0 • Sources: VS-KEYS |
| VS-WB-07 — Menus and contribution points | Main menus, editor/title actions, context menus, view actions, touch bars, and command-center entries are populated declaratively. | Place Velocity controls only where context makes them relevant: checkpoint actions in review editors, browser capture actions in browser tabs, and mission controls in project surfaces. | Use menu contribution points and custom workbench registries; maintain a menu placement standard.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT |
| VS-WB-08 — Primary and secondary side bars | Views and view containers can be moved between side bars or the panel, collapsed, reordered, and restored. | Keep the default stage sparse, but allow Explorer, Search, SCM, Outline, coworker roster, evidence, and activity to be docked by advanced users. | Contribute Velocity containers and preserve user layout customizations in workspace/profile storage.<br>Phase: P2 • Priority: P1 • Sources: VS-UI |
| VS-WB-09 — Panel and bottom drawer | Terminal, Problems, Output, Debug Console, Ports, and extension views share a dockable panel that can move or maximize. | Replace the prototype ToolDrawer with native panel containers. Velocity can add Logs, Checkpoints, Evidence, Decisions, Agent Runs, and Cost without duplicating terminal or Problems UI. | Register panel views and coordinate panel reveal through commands and attention policies.<br>Phase: P1 • Priority: P0 • Sources: VS-UI;VEL-01 |
| VS-WB-10 — Status bar | Compact left/right status entries show source-control state, language mode, diagnostics, remote context, feedback, and extension status. | Use the status bar for unobtrusive project health, active environment, autonomy mode, budget, trust, current coworker, and deployment state—never as a dashboard. | Register priority-ordered entries with accessible labels and click commands.<br>Phase: P2 • Priority: P1 • Sources: VS-UI |
| VS-WB-11 — Command center and title bar | A compact command center exposes search, navigation, workspace identity, and window actions while supporting custom title bars. | Adapt Velocity's quiet top bar into the native title/command-center layer so project identity, branch/environment, inbox, and account controls remain integrated with window management. | Product-level workbench customization; retain native drag regions and accessibility behavior.<br>Phase: P1 • Priority: P1 • Sources: VS-UI;VEL-01 |
| VS-WB-12 — Zen Mode and full-screen focus | Focus modes hide chrome and center the active editor while preserving escape paths and configurable restoration. | Velocity Focus Mode should be native Zen Mode plus project-specific suppression of coworker markers, rails, and secondary evidence until requested. | Extend Zen Mode configuration and context keys rather than building a separate overlay state machine.<br>Phase: P2 • Priority: P2 • Sources: VS-UI;VEL-02 |
| VS-WB-13 — Auxiliary windows | Editors and selected workbench content can move into additional native windows with independent placement. | Allow a live Preview, follow view, test dashboard, or evidence comparison to float on a second monitor while the main project remains visible. | Use auxiliary editor windows and serialized editor inputs; ensure agent sessions remain scoped to project/workstream.<br>Phase: P4 • Priority: P2 • Sources: VS-UI |
| VS-WB-14 — Layout profiles and restoration | Window, editor group, view, panel, and workspace state are serialized and restored across sessions. | Persist per-project Lens layouts, open evidence, browser sessions, terminal topology, and Follow Mode preferences while intentionally not restoring transient approvals or secrets. | Build a Velocity workspace-state schema on the native storage service with versioned migrations.<br>Phase: P2 • Priority: P1 • Sources: VS-SETTINGS;VEL-12 |
| VS-WB-15 — Notifications and progress | Transient notifications, progress indicators, status entries, and modal confirmations share standard services and accessibility behavior. | Route agent blockers, approvals, failed verification, conflicts, and deployment failures through an attention policy that escalates only material events. | Use notification/progress/dialog services with severity mapping, grouping, deduplication, and actionable commands.<br>Phase: P2 • Priority: P0 • Sources: VS-UI |
| VS-WB-16 — Breadcrumbs | Editors can display file path and symbol hierarchy, with keyboard navigation and quick selection. | Extend breadcrumbs with mission, checkpoint, candidate, semantic component, route, and scope-reservation context so users know where an artifact sits in the project. | Provide custom breadcrumb data or a compact adjacent project-context control where core APIs are insufficient.<br>Phase: P3 • Priority: P2 • Sources: VS-EDITOR |
| VS-WB-17 — Minimap and overview ruler | A scaled document preview and overview ruler surface diagnostics, search results, changes, breakpoints, and selections. | Add agent-change, reviewed/unreviewed, reservation, and evidence markers through decorations rather than inventing a separate gutter. | Use editor decorations, overview ruler lanes, and diff metadata.<br>Phase: P3 • Priority: P2 • Sources: VS-EDITOR |
| VS-WB-18 — Problems and diagnostics view | Language servers, linters, compilers, tasks, and extensions publish normalized diagnostics with navigation and filtering. | Treat diagnostics as first-class evidence. Checkpoints should snapshot counts, severity, new-versus-existing status, owner, and resolution trace. | Publish all verification findings through the marker service and index them into evidence records.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS;VS-EXT |
| VS-WB-19 — Output channels and logs | Services and extensions expose named, searchable output channels with log levels and timestamps. | Give each coworker, terminal, browser run, verifier, deployment, and orchestration service an output channel while surfacing only errors and requested traces by default. | Use log/output services with correlation IDs that link events to missions, work orders, and checkpoints.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| VS-WB-20 — Profiles | Profiles package settings, keybindings, snippets, UI state, tasks, and extensions for distinct workflows. | Offer Velocity profiles such as Product Owner, Design Review, Full IDE, QA, Incident, and Enterprise Locked Down. Profiles alter presentation, not project truth. | Build on native profiles and ship curated defaults; keep mission/coworker data outside profile scope.<br>Phase: P4 • Priority: P2 • Sources: VS-SETTINGS |
| VS-WB-21 — Settings Sync | User preferences, keybindings, snippets, tasks, profiles, and extensions can synchronize across signed-in devices. | Synchronize personal ergonomics while keeping repository instructions, enterprise policy, and project orchestration authoritative and auditable. | Reuse native sync or a Velocity-compatible service endpoint; explicitly classify syncable versus project-scoped data.<br>Phase: P7 • Priority: P3 • Sources: VS-SETTINGS |
| VS-WB-22 — Accessibility infrastructure | Workbench controls support keyboard operation, screen readers, high-contrast themes, reduced motion, zoom, accessible views, and audible signals. | All coworker presence, progress, decisions, evidence, and browser/terminal activity must have non-spatial text equivalents and complete keyboard flows. | Adopt VS Code accessibility services, signals, ARIA patterns, and automated accessibility regression tests.<br>Phase: P1 • Priority: P0 • Sources: VS-UI;VOSS-03 |
| VS-WB-23 — Theme and icon systems | Color themes, file/product icon themes, semantic token colors, and contributed CSS tokens allow broad personalization. | Keep Velocity's restrained visual identity as the default product theme while preserving user themes and high-contrast modes. Project state must never rely on color alone. | Define product colors as theme tokens, contribute iconography, and audit contrast in every built-in theme.<br>Phase: P2 • Priority: P1 • Sources: VS-UI;VEL-03 |
| VS-WB-24 — Walkthroughs and onboarding | Interactive walkthroughs connect documentation, settings, commands, and sample actions to product state. | Teach the autonomous-workspace model through a guided real project: create mission, observe isolated work, inspect evidence, redirect, accept checkpoint, and roll back. | Contribute walkthrough steps and instrument completion without fabricating agent activity.<br>Phase: P3 • Priority: P2 • Sources: VS-EXT |
| VS-WB-25 — Webviews and custom editors | Sandboxed webviews and custom editor providers host domain-specific experiences while participating in editor groups and lifecycle. | Use custom editors only for canvases, evidence viewers, data/schema explorers, and specialized visualization; keep files, terminal, SCM, tests, and browser native. | Build tightly sandboxed, CSP-safe custom editors with serializers, commands, and explicit data contracts.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT;VEL-03 |
