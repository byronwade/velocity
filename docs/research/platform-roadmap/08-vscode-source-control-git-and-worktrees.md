# 13. Source Control, Git, Branches, and Worktrees

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Git and native SCM provide the durable source-versioning layer for Stable, Candidate, work orders, checkpoints, review, merge, and rollback. Worktrees are the default isolation primitive for concurrent autonomous writes.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| SCM-01 — SCM provider model | Source-control providers contribute repositories, resource states, groups, commands, input boxes, history, decorations, and status. | Use native Git as the default provider and keep Velocity orchestration provider-neutral enough to support other SCM systems later. | Integrate through SCM services and Git extension APIs; do not parse Git porcelain in UI components.<br>Phase: P2 • Priority: P0 • Sources: VS-SCM |
| SCM-02 — Repository discovery | The workbench detects one or multiple repositories, nested repositories, submodules, and workspace roots, with configurable scanning. | A Velocity Project may contain several repos. Missions and checkpoints must record exactly which repository and revision each artifact belongs to. | Create a ProjectRepository registry and semantic root roles; block ambiguous cross-repo actions until scoped.<br>Phase: P2 • Priority: P0 • Sources: VS-SCM |
| SCM-03 — Working tree status | Tracked, untracked, modified, deleted, renamed, ignored, conflicted, and staged resources appear with decorations and grouping. | Surface agent changes through native SCM and annotate them with coworker/work-order identity, reviewed state, and checkpoint membership. | Add file decorations and SCM context metadata without replacing the Git view.<br>Phase: P2 • Priority: P0 • Sources: VS-SCM |
| SCM-04 — Stage and unstage | Users stage whole files, ranges, or selected changes and can discard or revert modifications. | Agent checkpoints should stage only their isolated worktree changes after validation. Human edits in the main working tree remain separate unless intentionally included. | Use Git worktrees and native staging; construct checkpoints from commits rather than shared index manipulation.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM;CUR-WT |
| SCM-05 — Commit workflow | Commit messages, amend, sign-off, signing, hooks, templates, and smart commit behavior are integrated into the workbench. | Create a machine-readable checkpoint commit with human-readable summary, mission/work-order IDs, evidence manifest hash, and model/tool provenance. | Use Git extension commands, repository hooks, signed commits where configured, and standardized trailers.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM;VEL-02 |
| SCM-06 — Diff and change navigation | Changed resources open in diff editors, support next/previous change, inline revert, stage selected ranges, and original-resource access. | Checkpoint review should open a curated multi-diff plus evidence, with files ordered by semantic importance and risk rather than alphabetically. | Build a checkpoint review editor that delegates file rendering to native diff editors.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM |
| SCM-07 — Branches | Users create, switch, rename, delete, publish, compare, and inspect branches with recent and remote branch information. | Map persistent coworker/candidate work to explicit branch policy. Branch names are implementation details beneath the human-facing mission/coworker identity. | Use Git branch operations with stable IDs stored in Velocity metadata; never rely on branch-name parsing alone.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM;VEL-02 |
| SCM-08 — Worktrees | Git worktrees provide isolated directories and branches for parallel work without duplicating object storage. | Make a worktree the default isolation boundary for every concurrent autonomous work order that can modify source. | Create, validate, lock, prune, and garbage-collect worktrees through a WorktreeService tied to agent sessions and checkpoints.<br>Phase: P4 • Priority: P0 • Sources: VS-WORKTREES;CUR-WT |
| SCM-09 — Stash and shelf | Uncommitted changes can be stashed, restored, popped, dropped, and inspected. | Avoid stashes as a primary agent mechanism because they are opaque and global to a repo. Use them only for human recovery or temporary handoff. | Expose native stash commands but keep autonomous work in commits/worktrees.<br>Phase: P3 • Priority: P2 • Sources: VS-SCM |
| SCM-10 — Remotes | Repositories manage fetch/push URLs, tracking branches, authentication, remote heads, pruning, and provider integrations. | Agents may fetch under policy but should not push, force-push, or change remotes without explicit permission. Remote actions are auditable protected operations. | Route remote commands through Git service, credential broker, network policy, and approval broker.<br>Phase: P4 • Priority: P0 • Sources: VS-SCM |
| SCM-11 — Fetch, pull, and synchronization | The workbench supports fetch, pull, rebase, push, synchronize, auto-fetch, and progress/error reporting. | A candidate should update from its base at controlled checkpoints, not unpredictably during work. Base drift becomes an explicit state and verification trigger. | Add base-revision tracking and update/rebase workflows to the work-order state machine.<br>Phase: P4 • Priority: P0 • Sources: VS-SCM;FIG-BRANCH |
| SCM-12 — Merge and rebase | Git merge/rebase operations, conflict states, continuation/abort, and editor integration are surfaced in the UI. | Use rebase/merge to refresh candidates and merge accepted checkpoints. Conflicts create Decision Sheets only when automation cannot resolve safely. | Drive native Git operations, merge editor, conflict diagnostics, and mandatory re-verification.<br>Phase: P4 • Priority: P0 • Sources: VS-SCM;VEL-02 |
| SCM-13 — Merge conflict editor | Conflict regions can be accepted individually or combined, with base/current/incoming/result context and completion validation. | Attach semantic ownership and test impact to conflicts. Agents may propose resolutions, but protected files or overlapping intent require human review. | Use native merge editor plus a conflict analyzer and reservation history.<br>Phase: P4 • Priority: P0 • Sources: VS-SCM |
| SCM-14 — History and graph | Commit history, file history, blame, branches, tags, comparisons, and remote metadata can be explored. | Unify Git history with Velocity checkpoints and deployment history. A checkpoint is a richer business object that references one or more commits. | Index commits into the event store and show mission/coworker/evidence metadata in custom history views.<br>Phase: P4 • Priority: P1 • Sources: VS-SCM |
| SCM-15 — Blame and authorship | Inline blame and file blame identify last commits/authors and can navigate history. | Distinguish human authorship, coworker identity, provider/model provenance, and supervising user without pretending the model is a legal author. | Standardize commit trailers and UI labels; respect repository authorship policies.<br>Phase: P5 • Priority: P1 • Sources: VS-SCM |
| SCM-16 — Tags and releases | Tags can mark versions, releases, deployment points, or milestones and be pushed to remotes. | Accepted release checkpoints may create signed tags after policy checks. Ordinary agent checkpoints should not create tags. | Integrate release workflow, signing, CI status, changelog, and deployment evidence.<br>Phase: P6 • Priority: P2 • Sources: VS-SCM |
| SCM-17 — Git hooks and policy checks | Client hooks can validate commits, messages, formatting, tests, secrets, or organization rules. | Agents must run repository hooks and cannot bypass them silently. Hook failures become evidence and block checkpoint readiness unless policy permits. | Run Git operations in normal environments, capture hook output, and record any approved bypass.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM |
| SCM-18 — Pull requests and code hosting | Provider extensions support creating PRs, reviewing comments, checks, issues, branches, remotes, and repository metadata. | Velocity should convert accepted candidate checkpoints into draft or ready PRs when the project uses review-based delivery, while preserving local-first operation. | Build provider adapters for GitHub/GitLab/etc.; keep core checkpoint model independent of hosting service.<br>Phase: P6 • Priority: P1 • Sources: VS-SCM;CUR-AWIN |
| SCM-19 — Reviewed-file state | Agent review systems can distinguish files a human has inspected from files still changing or unreviewed. | Track review at file and hunk level per checkpoint revision. New agent edits invalidate review only for affected ranges and dependent evidence. | Store review markers against commit/blob/range identities, not mutable working files.<br>Phase: P5 • Priority: P0 • Sources: VS-AGENTSW;VOSS-01 |
| SCM-20 — Scope reservations | Velocity defines temporary ownership of files, symbols, routes, schemas, or feature areas to reduce agent collisions. | Reservations should complement—not replace—Git isolation. They provide coordination, conflict prediction, and user visibility while worktrees preserve correctness. | Implement reservation service over semantic graph IDs with leases, overlap detection, priority, and release events.<br>Phase: P5 • Priority: P0 • Sources: VEL-02;FIG-BRANCH |
| SCM-21 — Checkpoint commit set | One checkpoint can span multiple repositories or commits but must resolve to an immutable, verifiable source state. | Create a manifest listing base/head commits, repositories, evidence, environment, approvals, and rollback targets. Acceptance atomically advances Stable at the product layer. | Define signed checkpoint manifests and an acceptance transaction with compensation on partial failure.<br>Phase: P5 • Priority: P0 • Sources: VEL-02 |
| SCM-22 — Rollback and revert | Git supports revert commits, branch reset, checkout of historical files, and new branches from prior revisions. | Velocity rollback should choose the safest method based on whether a checkpoint is local, merged, deployed, or externally depended upon. | Implement rollback plans that preview source, data, environment, and deployment effects before execution.<br>Phase: P5 • Priority: P0 • Sources: VEL-02 |
