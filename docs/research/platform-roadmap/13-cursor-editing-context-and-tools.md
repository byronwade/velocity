# 18. Cursor AI Editing, Context, and Interaction Patterns

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Cursor demonstrates a mature interaction model for autonomous and assisted editing: agent, ask, plan, context retrieval, steering, checkpoints, review, and model selection. Velocity should preserve these capabilities while moving durable state out of chat.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| CUR-EDIT-01 — Agent mode | Cursor's Agent independently explores a codebase, searches files, edits files, runs commands, uses browser tools, and iterates toward a result. | Velocity should use autonomous sessions as work orders beneath named coworkers. The visible state is the artifact/evidence, not a permanent agent transcript. | Provider adapter plus ToolBroker and work-order state machine.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-02 — Ask/read-only interaction | A non-editing mode answers questions, investigates code, and gathers context without changing files. | Provide an Inspect action available from any artifact, file, test, browser element, or checkpoint; it must guarantee no write/execute capabilities. | Issue a read-only capability envelope and enforce it at tool authorization, not only prompt text.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-03 — Plan mode | The agent investigates and produces a structured plan before editing, allowing review and iteration. | Velocity missions should compile into plans with scope, criteria, risk, staffing, dependencies, and evidence requirements before spawning work orders. | Create typed Plan and PlanStep resources and a plan-review editor.<br>Phase: P4 • Priority: P0 • Sources: CUR-PLAN |
| CUR-EDIT-04 — Inline edit | Natural-language edits can be requested in the current file or selection and reviewed in context. | Offer contextual 'Direct this' commands in takeover mode, but route broader changes into isolated work orders with diffs. | Use inline chat/edit APIs for small scopes; promote to checkpoint workflow above thresholds.<br>Phase: P4 • Priority: P1 • Sources: CUR-AGENT |
| CUR-EDIT-05 — Tab completions | AI predicts code at the cursor, including multiline completions and edits informed by local context. | Keep optional for human editing; separate telemetry and settings from autonomous coworker execution. | Use a provider-neutral inline completion adapter with attribution and enterprise controls.<br>Phase: P5 • Priority: P2 • Sources: CUR-AGENT |
| CUR-EDIT-06 — Next-edit prediction | The editor predicts where and how the user is likely to edit next after accepting a change. | Useful only in hands-on mode. It should never move focus during review or while following an agent. | Integrate through native next-edit suggestions with context gating.<br>Phase: P5 • Priority: P3 • Sources: CUR-AGENT |
| CUR-EDIT-07 — Codebase indexing | Cursor builds an index of repository content to retrieve relevant files and symbols for AI context. | Velocity needs a multi-layer context index: lexical, semantic symbols, dependency graph, Git history, diagnostics, tests, routes, design tokens, evidence, and user instructions. | Implement incremental local-first indexing with optional embeddings and strict project boundaries.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-EDIT-08 — Automatic context selection | The agent chooses relevant files, symbols, terminal output, web content, and other context rather than requiring manual attachment of everything. | Use retrieval policies based on work-order scope, semantic graph, current failures, and token budget; every context item carries provenance and revision. | Build ContextBroker with ranking, deduplication, summarization, and disclosure in run details.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-09 — Manual context attachments | Users can attach files, folders, symbols, code ranges, docs, images, terminal output, web pages, or previous conversations. | Velocity replaces chat attachments with artifact-level references: pins, selected semantic scopes, evidence items, and mission resources. | Provide a universal ResourceRef model and contextual 'Add to mission/work' actions.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-EDIT-10 — Image input | Images or screenshots can guide UI implementation, debugging, or visual comparison. | Store images as source-linked artifacts with rights/provenance and use them in design or verification work orders, not as untraceable clipboard blobs. | Add artifact ingestion, metadata, OCR-free visual context, and redaction controls.<br>Phase: P4 • Priority: P1 • Sources: CUR-AGENT |
| CUR-EDIT-11 — Web search and documentation lookup | The agent can research external sources and use web content as context. | Make research an explicit tool with domain policy, source capture, citation requirements, and separation from code execution. | Implement WebResearchTool with first-party preference, snapshot metadata, and evidence links.<br>Phase: P5 • Priority: P1 • Sources: CUR-AGENT |
| CUR-EDIT-12 — Documentation context | Product/library documentation can be indexed or fetched so the agent uses version-relevant APIs. | Resolve documentation from lockfiles, package versions, language servers, local docs, and approved web sources; avoid generic memory when exact version matters. | Create a dependency-to-doc resolver and versioned source cache.<br>Phase: P5 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-13 — Rules | Cursor supports project and user rules that inject persistent instructions based on scope or activation. | Map rules into organization, project, path/language, mission, and coworker scopes with explicit precedence and version control. | Create `.velocity/rules` schema, rule inspector, conflict detection, and provider export adapters.<br>Phase: P5 • Priority: P0 • Sources: CUR-RULES |
| CUR-EDIT-14 — Memories | The system can retain useful user/project facts for later sessions. | Durable memory must be visible, source-backed, scoped, editable, expiring, and excluded from unrelated projects by default. | Implement typed memory records and a memory management view.<br>Phase: P5 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-15 — Queued messages | Users can queue additional instructions while an agent is working, reorder them, or interrupt with urgent direction. | Velocity should support queued directives on a work order without forcing chat. Pins, revision notes, and decisions become ordered control events. | Add directive queue with priority, supersession, and safe-point application semantics.<br>Phase: P5 • Priority: P0 • Sources: CUR-AGENT |
| CUR-EDIT-16 — Ask while working | A user can ask the agent a question without necessarily stopping its current task. | Follow Mode should provide a contextual question/action panel whose read-only questions do not mutate the work order unless converted into a directive. | Separate inquiry events from control events in the session protocol.<br>Phase: P5 • Priority: P1 • Sources: CUR-AGENT |
| CUR-EDIT-17 — Interrupt and steer | The agent can be stopped or redirected during execution while preserving progress. | Provide pause-at-safe-point, immediate cancel, revise scope, change criteria, and resume-from-checkpoint controls. | Implement cancellation tokens, state snapshots, tool interruption, and directive reconciliation.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-EDIT-18 — Checkpoints | Cursor can save intermediate agent states and let users restore earlier points. | Velocity checkpoints must be repository- and evidence-backed, coherent across files and tools, and suitable for acceptance or rollback. | Create immutable checkpoint manifests and review UI; use session checkpoints only as internal recovery.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-EDIT-19 — Review changes | A dedicated review surface shows agent edits, supports accepting/rejecting hunks or files, and keeps review state. | Use native multi-diff with file/hunk reviewed state, semantic ordering, evidence side panel, and invalidation when new changes arrive. | Build checkpoint review editor over SCM/diff services.<br>Phase: P4 • Priority: P0 • Sources: CUR-AWIN;VS-AGENTSW |
| CUR-EDIT-20 — Model selection | Cursor exposes multiple models and automatic selection based on task/capability. | Velocity selects provider/model beneath coworker identity using policy for quality, privacy, latency, cost, modality, context, and tool support. | Implement ModelRouter with capability descriptors, fallback, budget, and per-work-order overrides.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-EDIT-21 — Long-context handling | Agents can use large context windows but still rely on retrieval, summaries, and tool calls to avoid loading entire repositories. | Velocity should optimize for evidence-grounded incremental context rather than equating more tokens with better reasoning. | Add token budgets, context manifests, compression, and retrieval evaluations.<br>Phase: P5 • Priority: P1 • Sources: CUR-AGENT |
| CUR-EDIT-22 — Conversation/session history | Prior turns and tool activity are retained for continuity and review. | Store an optional audit/session history, but do not make it the primary product surface or expose hidden reasoning. | Persist user-visible messages, plans, tool calls, outputs, decisions, and summaries with retention controls.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |

# 19. Cursor Agent Tools and Execution Mechanics

Cursor's file, search, terminal, browser, design, MCP, and tool lifecycle patterns inform the ToolBroker contract. Velocity must strengthen them with source authority, native Code-OSS integration, policy, evidence, and work-order isolation.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| CUR-TOOLS-01 — File read tool | Agents can open and read files or selected ranges as needed. | Use URI/revision-aware reads with size limits, binary detection, secret policy, and provenance; prefer semantic queries over bulk reads. | Wrap native text/file services in a typed ReadResourceTool.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-TOOLS-02 — File write tool | Agents can create, modify, rename, or delete files and present edits for review. | All writes occur in isolated worktrees through transactional workspace edits; protected paths, generated files, and ownership rules are enforced. | Implement WorkspaceEdit transactions plus Git snapshots and policy checks.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-TOOLS-03 — Search tool | Agents search file names, content, symbols, and repository structure to locate relevant code. | Unify lexical search, file search, symbol search, references, graph traversal, diagnostics, and history under query-specific tools. | Build SearchBroker over native search/language/graph services.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-TOOLS-04 — Terminal tool | Agents execute commands, read output, respond to prompts, and iterate based on exit status. | Use native PTYs and tasks under per-work-order policies, not a simulated shell or unconstrained global terminal. | Implement AgentTerminalService over Code-OSS terminal/process APIs.<br>Phase: P4 • Priority: P0 • Sources: CUR-TERM;VOSS-03 |
| CUR-TOOLS-05 — Terminal foreground mode | Commands run visibly and may require the agent to wait for completion or interact with output. | Use for finite builds, migrations, tests, and scripts; capture command boundaries and structured results. | Map to terminal/task execution with timeout, cancellation, and evidence.<br>Phase: P4 • Priority: P0 • Sources: CUR-TERM |
| CUR-TOOLS-06 — Terminal background mode | Long-running servers or watchers continue while the agent performs other work. | Track readiness, owner, port, logs, resource usage, and cleanup; background processes cannot outlive policy silently. | Create ManagedProcess entities tied to work orders and service registry.<br>Phase: P4 • Priority: P0 • Sources: CUR-TERM |
| CUR-TOOLS-07 — Terminal sandbox mode | Agent commands can be restricted to a sandbox that limits filesystem and network access. | Offer OS/container/worktree sandbox backends with explicit capability profiles; prompt-only restrictions are insufficient. | Build execution adapters for local sandbox runtime, containers, remote runners, and enterprise policy.<br>Phase: P5 • Priority: P0 • Sources: CUR-TERM;VOSS-01 |
| CUR-TOOLS-08 — Browser navigation tool | Agents open URLs, navigate, follow links, and maintain session state. | Bind navigation to approved domains and the correct project/agent browser partition. | Use integrated browser/CDP tools through the ToolBroker.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW;VS-BROWSER |
| CUR-TOOLS-09 — Browser read tool | Agents inspect page text, DOM structure, accessibility information, console output, and page state. | Return structured snapshots with locators and revision/session provenance rather than full raw HTML by default. | Create BrowserSnapshotTool with bounded detail levels.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-10 — Browser interaction tool | Agents click, type, scroll, hover, drag, choose options, upload files, and handle dialogs. | Use resilient accessible locators, explicit waits, retry budgets, and action evidence; sensitive interactions require approvals. | Wrap Playwright actions with locator strategy and policy metadata.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-11 — Browser screenshot tool | Agents capture viewport, element, area, or full-page images to understand and verify UI. | Store deterministic screenshot artifacts and compare against Stable or expected references. | Build capture/evidence pipeline with environment metadata.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-12 — Browser console tool | Agents inspect runtime errors, warnings, logs, and stack traces. | Automatically correlate console findings to source maps and checkpoint revisions. | Subscribe through CDP and index normalized console events.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-13 — Browser network tool | Agents inspect request failures, API traffic, timing, payload metadata, and external dependencies. | Use network traces for behavioral evidence, security policy, and performance regression detection. | Capture redacted HAR/CDP events with domain and schema analysis.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-14 — Playwright execution | Agents can run Playwright code for complex browser workflows beyond primitive tools. | Allow generated scripts in a constrained test harness, persist the script, and record all actions/assertions; forbid opaque unreviewed production automation. | Create ephemeral Playwright test files or virtual artifacts with sandbox and trace.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW;VS-BROWSER |
| CUR-TOOLS-15 — Design manipulation sidebar | Cursor's browser design tools can select elements and change layout, dimensions, colors, and themes across one or multiple elements. | Translate this into semantic intent controls. The user may select regions or component families, but the agent modifies source and design tokens; no hidden DOM-only patch becomes product truth. | Resolve selection to source graph, generate source patch, rebuild, and verify across routes/viewports.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW;VEL-11 |
| CUR-TOOLS-16 — Multi-select design edits | Multiple page elements can be selected and adjusted together, including parallel updates. | Velocity should support 'apply this intent across all matching components/routes' using semantic graph scopes and design-system rules. | Use component identity, token usage graph, codemods, and visual regression matrix.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW |
| CUR-TOOLS-17 — Image generation tool | Agents can generate image assets and place them into a project. | Treat generated assets as new source files with prompt/model/license metadata, review, optimization, and responsive usage—not transient browser pixels. | Add MediaGenerationTool, asset pipeline, provenance manifest, and rights controls.<br>Phase: P6 • Priority: P1 • Sources: CUR-AGENT |
| CUR-TOOLS-18 — MCP tools | Agents can call external Model Context Protocol servers for databases, docs, issue trackers, design systems, browsers, and custom services. | Support MCP as an extension path, but wrap every server through policy, typed capability discovery, secret brokering, and audit. Core IDE/browser/SCM tools remain native. | Build an MCP registry and gateway with per-project allowlists and tool-risk metadata.<br>Phase: P5 • Priority: P0 • Sources: CUR-MCP |
| CUR-TOOLS-19 — Tool result rendering | Tool calls can show progress, status, outputs, links, diffs, and errors inside the agent experience. | Render results as artifact updates, evidence cards, native views, and timelines. The session detail may show tool events, but the main stage should not become a tool transcript. | Create standard ToolEvent and ArtifactRef renderers across workbench surfaces.<br>Phase: P4 • Priority: P1 • Sources: CUR-AGENT;VEL-02 |
| CUR-TOOLS-20 — Tool cancellation | Long-running or blocked tools can be interrupted when a user stops or redirects an agent. | Cancellation must propagate to PTYs, tasks, browsers, debug sessions, network calls, MCP servers, subagents, and remote runners. | Use hierarchical cancellation tokens and cleanup contracts.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-TOOLS-21 — Tool retries | Agents can retry failed commands or interactions, often adapting parameters based on output. | Define bounded retry policies by tool/error class and record each attempt; never loop indefinitely or hide repeated failure. | Implement retry budgets, backoff, deduplication, and escalation triggers.<br>Phase: P4 • Priority: P0 • Sources: CUR-AGENT |
| CUR-TOOLS-22 — Tool approval memory | Users may approve a class of safe actions for the current session or workspace instead of confirming every call. | Represent approvals as explicit capability leases with scope, pattern, expiry, owner, and revocation—not vague 'always allow' state. | Build CapabilityLease and policy matching service.<br>Phase: P4 • Priority: P0 • Sources: CUR-TERM;CUR-APPROVAL |
| CUR-TOOLS-23 — Tool context continuity | A terminal, browser page, file set, or test run remains associated with the active agent session across steps. | Propagate immutable session/work-order IDs through every native service so evidence and permissions cannot cross sessions accidentally. | Add AgentExecutionContext to service factories and events.<br>Phase: P4 • Priority: P0 • Sources: CUR-AWIN |
| CUR-TOOLS-24 — Tool environment variable | Cursor exposes indicators such as `CURSOR_AGENT` so commands can detect agent execution. | Expose standardized, non-secret metadata such as `VELOCITY_AGENT`, work-order ID, project ID, worktree root, and policy profile for scripts and hooks. | Inject minimal environment variables and a richer local metadata endpoint; document stability.<br>Phase: P4 • Priority: P1 • Sources: CUR-TERM |
| CUR-TOOLS-25 — Agent-ready tool contract | A tool is safe for autonomy only when its input/output schema, permissions, cancellation, idempotency, side effects, provenance, and evidence behavior are defined. | Use this as the acceptance bar for every Velocity tool, including project-specific integrations. | Publish a Tool SDK and automated conformance suite.<br>Phase: P5 • Priority: P0 • Sources: CUR-AGENT;VS-AGENTS |
