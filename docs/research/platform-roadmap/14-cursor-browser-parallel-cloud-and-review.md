# 20. Cursor Browser, Design, and Canvas Patterns

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Cursor's browser design controls and persistent Canvas support agent-driven visual work. Velocity should translate those controls into semantic source edits, multi-route verification, and durable project artifacts.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| CUR-VIS-01 — Browser testing loop | Cursor agents can launch an app, navigate flows, inspect failures, edit code, and rerun browser checks. | Make this the default UI verification loop for Velocity: start candidate, exercise acceptance flows, collect evidence, fix, repeat within bounded cycles. | Orchestrate tasks, ports, browser sessions, source maps, and checkpoint evidence.<br>Phase: P4 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-02 — Visual edit mode | The browser exposes selected-element controls for layout, size, spacing, color, and theme changes that the agent applies. | Velocity should present high-level intent knobs and semantic selections, then produce source edits across components/tokens. Any browser-only mutation is a temporary preview until committed to source. | Build selection-to-source mapping, design intent schema, and patch/verify pipeline.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-03 — Element inspection | Users or agents can select rendered elements and inspect structure, style, dimensions, and relationships. | Capture DOM, accessibility node, computed style, box model, event/listener hints, screenshots, and source links as one ElementEvidence object. | Integrate CDP, source maps, framework instrumentation, and semantic graph.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-04 — Parallel visual changes | Several selected elements or routes can be modified in one operation, with the agent coordinating the edits. | Support scope expansion from instance to component, variant, route family, design token, or entire product; show affected surfaces before execution. | Implement semantic scope resolver and impact preview.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-05 — Layout controls | Visual tools can adjust display, flex/grid behavior, alignment, gaps, sizing, position, padding, and margins. | Translate controls to project-native source: CSS, utility classes, component props, tokens, or framework-specific styles, with a preview patch. | Create framework adapters and style provenance resolver; never normalize styles blindly.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-06 — Dimension controls | Visual tools can set width, height, min/max sizes, responsive constraints, and aspect ratio. | Apply intent across breakpoint policy and component semantics rather than hard-coding one viewport. | Add responsive rule inference and multi-viewport verification.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-07 — Color and theme controls | Visual tools can change foreground/background/border colors and switch or create themes. | Prefer design tokens and variables; propose token creation or alias changes when raw values would increase drift. | Use token graph, contrast checks, theme matrices, and source diffs.<br>Phase: P6 • Priority: P0 • Sources: CUR-BROW;FIG-VARS |
| CUR-VIS-08 — Typography controls | A design surface can adjust font family, size, weight, line height, letter spacing, alignment, and text styles. | Map changes to typography tokens/components and verify overflow, wrapping, localization, and accessibility across routes. | Build typography token adapters and visual/a11y tests.<br>Phase: P6 • Priority: P1 • Sources: FIG-DEV |
| CUR-VIS-09 — Responsive preview | The browser can inspect and edit across desktop/mobile contexts. | Velocity should run named viewport/device matrices and show Stable/Candidate evidence side by side, including interaction—not only static frames. | Use browser profiles and synchronized route capture.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW;VEL-11 |
| CUR-VIS-10 — Console-aware design iteration | Visual work is informed by console errors and runtime state, preventing design changes from being evaluated as static images only. | A design checkpoint is not ready if it introduces runtime errors, hydration issues, warnings, or failed requests. | Make console/network/interaction gates mandatory for UI work.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW |
| CUR-VIS-11 — Accessibility-aware design iteration | Browser inspection can surface accessibility structure and support accessible locators. | Every visual change should run contrast, name/role/state, keyboard path, focus, reduced-motion, and zoom checks as applicable. | Integrate axe/browser accessibility APIs and manual review hooks.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW;FIG-CHECK |
| CUR-VIS-12 — Canvas artifact | Cursor Canvas creates persistent interactive artifacts that can be saved, reopened, rerun, and refined with the agent. | Use a Canvas editor for durable system maps, workflow diagrams, UI alternatives, data visualizations, and design explorations, each backed by source or a portable document. | Implement a custom editor with artifact schema, source links, render runtime, and checkpointing.<br>Phase: P6 • Priority: P1 • Sources: CUR-CANVAS |
| CUR-VIS-13 — Agent-first canvas refinement | Users describe changes and the agent edits the artifact rather than requiring manual low-level authoring. | This matches Velocity's direction: users direct outcomes and constraints; agents update the underlying source/graph. Manual direct editing is optional only for high-level layout/annotation controls. | Define semantic canvas operations and agent tool contracts.<br>Phase: P6 • Priority: P0 • Sources: CUR-CANVAS;VEL-02 |
| CUR-VIS-14 — Canvas source persistence | Canvas artifacts remain addressable and can be rerun or published instead of disappearing in a transient chat response. | Store artifacts in the project or evidence store with versions, dependencies, renderer, and export formats. | Create `*.velocity-canvas.json` or equivalent portable format plus custom editor.<br>Phase: P6 • Priority: P1 • Sources: CUR-CANVAS |
| CUR-VIS-15 — Canvas interactivity | Artifacts can include controls, diagrams, data views, prototypes, or executable visualizations rather than static images. | Sandbox executable artifacts and distinguish trusted project code from generated review visualizations. | Use sandbox runtime/webview with CSP, permissions, and deterministic data inputs.<br>Phase: P6 • Priority: P0 • Sources: CUR-CANVAS;VOSS-01 |
| CUR-VIS-16 — Canvas publication | Artifacts can be shared or published for collaborators. | Publish read-only snapshots with revision/evidence identity, expiration, access policy, and comments; avoid making an unpublished generated canvas a product dependency. | Add artifact sharing service and static export pipeline.<br>Phase: P7 • Priority: P2 • Sources: CUR-CANVAS |
| CUR-VIS-17 — Design alternatives | Agents can produce and compare several visual approaches before applying one. | Represent alternatives as candidate branches/checkpoints, with synchronized routes and explicit tradeoffs, not untracked generated screenshots. | Use temporary worktrees, browser capture matrix, and Decision Sheet.<br>Phase: P6 • Priority: P1 • Sources: VEL-02;FIG-BRANCH |
| CUR-VIS-18 — Browser-to-mission pin | A user can point at a rendered location and assign work without manually locating source. | Enhance existing Velocity comment pins with DOM/accessibility/source semantic identity so pins survive layout changes and route reloads. | Store locator sets, source graph IDs, screenshot crop, coordinates as fallback, and revision.<br>Phase: P5 • Priority: P0 • Sources: VEL-01;VEL-12 |
| CUR-VIS-19 — Whole-product transformation | The user may request a change such as 'apply the new visual system everywhere' and let the agent coordinate cross-cutting edits. | Resolve the request into tokens, components, routes, docs, tests, and migration plan; preview blast radius and stage checkpoints rather than one opaque giant edit. | Use semantic graph, dependency analysis, subagents, worktrees, and incremental checkpoints.<br>Phase: P6 • Priority: P0 • Sources: VEL-02 |
| CUR-VIS-20 — Visual feature acceptance gate | A feature is complete only when source changes reproduce in clean builds, all named viewports/flows pass, visual deltas are explainable, accessibility and console/network gates pass, and rollback works. | This gate prevents browser-only visual manipulation or screenshot theater from being treated as shipped work. | Automate a VisualVerificationPipeline and enforce it in checkpoint readiness.<br>Phase: P5 • Priority: P0 • Sources: CUR-BROW;FIG-CHECK;VEL-02 |

# 21. Cursor Parallel Agents, Cloud Execution, Automation, and Review

Agents Window, worktrees, cloud VMs, remote browser/desktop, automations, Bugbot, approval agents, and fan-out provide the reference set for scaling beyond one local session.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| CUR-SCALE-01 — Agents Window | Cursor provides a dedicated window for creating, monitoring, switching, and reviewing multiple agent sessions across repositories and environments. | Velocity should preserve the session-management capability but present coworkers and progress in the project stage by default, with a dense manager view on demand. | Use native/workbench session services and a Velocity coworker/work-order projection.<br>Phase: P5 • Priority: P0 • Sources: CUR-AWIN |
| CUR-SCALE-02 — Parallel local agents | Several agents can work simultaneously, each with isolated changes, terminals, and context. | Use one Git worktree and execution namespace per write-capable work order, with semantic reservations to predict overlaps. | Implement scheduler, worktree service, resource quotas, and conflict analyzer.<br>Phase: P5 • Priority: P0 • Sources: CUR-AWIN;CUR-WT |
| CUR-SCALE-03 — Worktree isolation | Cursor creates Git worktrees so parallel agents do not overwrite one another's files. | Adopt as the default source isolation mechanism for autonomous writes; a shared checkout is reserved for human takeover or explicitly coordinated sessions. | Build robust create/reuse/lock/prune/recover operations and map worktrees to sessions.<br>Phase: P4 • Priority: P0 • Sources: CUR-WT |
| CUR-SCALE-04 — Worktree change review | Agent worktree diffs can be reviewed, committed, merged, or turned into pull requests. | Wrap this in Velocity checkpoints with evidence and an accept/revise/reject decision before Stable advances. | Use native SCM/multi-diff plus checkpoint manifest and merge transaction.<br>Phase: P5 • Priority: P0 • Sources: CUR-AWIN;VEL-02 |
| CUR-SCALE-05 — Cross-repository sessions | Agents Window can operate across multiple repositories and workspaces. | Support a single mission spanning frontend, backend, infrastructure, docs, or design-system repos, but issue separate scoped work orders and commits per repo. | Create cross-repo mission coordinator and atomic checkpoint manifest.<br>Phase: P6 • Priority: P1 • Sources: CUR-AWIN |
| CUR-SCALE-06 — SSH environments | Agents can run against remote SSH workspaces and tools. | Treat SSH as an execution authority with its own trust, credentials, worktrees, terminals, ports, and cleanup. | Reuse Code-OSS Remote SSH and AgentExecutionContext.<br>Phase: P6 • Priority: P1 • Sources: CUR-AWIN;VS-REMOTE |
| CUR-SCALE-07 — Cloud agents | Agents execute in managed remote virtual machines with repository access, tools, browser, and artifacts. | Add pluggable RemoteExecutionProviders for Velocity Cloud, customer VPC, or third-party runners, while preserving the same work-order/evidence protocol. | Define runner protocol, image/environment manifests, encrypted credentials, and event streaming.<br>Phase: P7 • Priority: P1 • Sources: CUR-CLOUD |
| CUR-SCALE-08 — Cloud VM lifecycle | Cloud sessions provision compute, clone repositories, install dependencies, run tools, then suspend or destroy resources. | Tie compute lifecycle to work orders, budgets, idle timeouts, checkpoints, and audit; cleanup failure becomes an operational alert. | Build runner orchestrator, lease manager, snapshot/cache policy, and teardown verifier.<br>Phase: P7 • Priority: P0 • Sources: CUR-CLOUD |
| CUR-SCALE-09 — Remote browser capability | Cloud agents can use browser automation and remote desktop capabilities inside their environment. | Route browser evidence from the remote authority and offer read-only live view/handoff without exposing unrelated sessions. | Implement remote CDP/Playwright transport, video/trace streaming, and scoped control leases.<br>Phase: P7 • Priority: P1 • Sources: CUR-CLOUD |
| CUR-SCALE-10 — Remote desktop handoff | A human can inspect or take control of a cloud agent environment when automation is insufficient. | Provide explicit handoff with session freeze/ownership transfer, visible recording/audit state, and controlled clipboard/file transfer. | Build remote desktop gateway and capability transition workflow.<br>Phase: P7 • Priority: P1 • Sources: CUR-CLOUD |
| CUR-SCALE-11 — Artifacts from cloud runs | Remote agents return files, screenshots, logs, plans, builds, or reports as durable artifacts. | Normalize all remote outputs into the same ArtifactRef/evidence model as local runs and verify hashes before use. | Create artifact upload protocol, content-addressed storage, scanning, and retention.<br>Phase: P7 • Priority: P0 • Sources: CUR-CLOUD |
| CUR-SCALE-12 — Environment snapshots and caches | Cloud execution can reuse dependency caches, machine images, or prepared environments to reduce startup time. | Cache by trusted environment manifest, lockfile, toolchain, architecture, and policy; never reuse mutable credentials or browser state unintentionally. | Implement signed image/cache keys and cache-poisoning protections.<br>Phase: P7 • Priority: P1 • Sources: CUR-CLOUD |
| CUR-SCALE-13 — Automations | Agents can be scheduled or triggered by events such as commits, pull requests, issues, or external systems. | Velocity automations should create visible missions/work orders under policy rather than silently mutate projects. Examples: dependency updates, nightly tests, visual audits, documentation drift. | Build trigger service, automation definitions, rate limits, approvals, and owner/inbox routing.<br>Phase: P7 • Priority: P1 • Sources: CUR-AUTO |
| CUR-SCALE-14 — Scheduled agents | Time-based agents run at recurring intervals or one-off times. | Use for maintenance and evidence generation; write-capable schedules default to draft checkpoints, never automatic Stable changes unless explicitly governed. | Implement cron scheduler and policy-enforced output mode.<br>Phase: P7 • Priority: P1 • Sources: CUR-AUTO |
| CUR-SCALE-15 — Event-triggered agents | Repository, CI, monitoring, issue, or webhook events can launch targeted agents. | Translate events into typed mission templates with deduplication, correlation, and suppression to avoid incident storms. | Create event router, idempotency keys, trigger filters, and escalation policy.<br>Phase: P7 • Priority: P0 • Sources: CUR-AUTO |
| CUR-SCALE-16 — Bugbot pull-request review | Cursor's Bugbot reviews pull requests for likely defects and posts findings. | Implement review coworkers that analyze diffs, tests, runtime evidence, security, and project rules, then publish evidence-backed findings with confidence and reproducibility. | Use diff scanners, language/graph context, test evidence, and hosting-provider comments.<br>Phase: P7 • Priority: P1 • Sources: CUR-BUGBOT |
| CUR-SCALE-17 — Approval agents | Specialized agents evaluate requests against policy and can approve, deny, or escalate. | Use approval agents only as advisory/policy evaluators with explicit authority bounds. High-impact production or governance decisions still follow organization policy and human ownership. | Build ApprovalPolicy, evaluator plugins, explanations, and mandatory audit.<br>Phase: P7 • Priority: P0 • Sources: CUR-APPROVAL |
| CUR-SCALE-18 — Automated code review | Agents can inspect diffs for bugs, style, maintainability, tests, security, or regressions. | Create independent reviewer work orders that cannot modify the candidate they review and whose findings link to exact ranges/evidence. | Run in readonly clones/snapshots with model diversity and validation rules.<br>Phase: P6 • Priority: P0 • Sources: CUR-BUGBOT |
| CUR-SCALE-19 — Agent handoff | A local session can hand work to a cloud agent or another environment and later resume/review results. | Serialize plan, context manifest, base revision, work order, tool state, and artifacts rather than relying on an opaque transcript. | Define portable SessionBundle format and compatibility negotiation.<br>Phase: P7 • Priority: P1 • Sources: CUR-AWIN |
| CUR-SCALE-20 — Subagent fan-out | A coordinator can dispatch several agents for research, implementation, testing, review, or alternatives. | Represent each child as a Specialist with isolated scope, budget, and return contract; parent synthesis is itself reviewable. | Implement DAG scheduler, dependency states, cancellation propagation, and merge/conflict strategy.<br>Phase: P6 • Priority: P0 • Sources: CUR-SUB |
| CUR-SCALE-21 — Resource budgeting | Parallel/cloud agents consume tokens, CPU, memory, storage, network, browser sessions, and external API quotas. | Expose project and mission budgets, predicted/actual usage, throttling, and hard stops. Cost is evidence, not a fake progress meter. | Build unified BudgetService and per-tool metering.<br>Phase: P5 • Priority: P0 • Sources: CUR-CLOUD;VEL-02 |
| CUR-SCALE-22 — Failure recovery | Sessions may crash, lose network, time out, or encounter unavailable models/tools. | Persist state at tool boundaries, retry only idempotent steps, resume from checkpoint, fail over providers, and surface unrecoverable blockers. | Implement durable orchestration log and recovery coordinator.<br>Phase: P5 • Priority: P0 • Sources: CUR-AWIN |
| CUR-SCALE-23 — Parallel conflict prediction | Concurrent agents may touch overlapping files, symbols, schemas, APIs, routes, or design tokens. | Use semantic reservations, dependency graph, planned edits, and live diffs to warn or serialize before conflicts materialize. | Build overlap scoring and negotiation events between work orders.<br>Phase: P6 • Priority: P0 • Sources: VEL-02;FIG-BRANCH |
| CUR-SCALE-24 — Merge orchestration | Completed parallel work must be ordered, rebased, tested together, and merged without invalidating evidence. | Construct an integration candidate, replay checkpoints in dependency order, resolve conflicts, and rerun combined verification before Stable advances. | Create IntegrationWorkOrder and merge queue.<br>Phase: P6 • Priority: P0 • Sources: CUR-WT;VEL-02 |
| CUR-SCALE-25 — Parallel-agent acceptance gate | Production readiness requires isolation, quotas, cancellation, conflict handling, recovery, combined verification, review-state invalidation, and deterministic cleanup across local and remote sessions. | Do not market 'multiple coworkers' as real until parallel write-capable sessions pass this gate under failure and restart scenarios. | Build chaos/end-to-end tests for overlapping edits, crashed runners, stale bases, leaked ports, and revoked credentials.<br>Phase: P6 • Priority: P0 • Sources: CUR-AWIN;CUR-CLOUD;VEL-12 |
