# 16. Extensions, Remote Development, Notebooks, Accessibility, and Enterprise

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Velocity's long-term leverage depends on Code-OSS ecosystem compatibility, remote authorities, extensibility, product distribution, accessibility, policy, and audit. These cannot be bolted on after the agent UI is complete.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| PLATFORM-01 — Extension host isolation | Extensions execute in extension hosts rather than the renderer, with local, remote, web, and utility-process placement depending on capabilities. | Build Velocity product features as workbench/core contributions where deep integration is required, and expose an extension SDK for optional integrations without giving every extension orchestration privileges. | Retain extension-host architecture; define a Velocity API namespace with explicit capability grants.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| PLATFORM-02 — Extension manifests | Extensions declare activation events, engines, capabilities, dependencies, contribution points, commands, settings, languages, views, and permissions through `package.json`. | Use manifest-driven discovery for Velocity tools, evidence producers, coworker skills, design analyzers, deploy adapters, and data connectors. | Extend manifest schemas with versioned Velocity contributions while keeping ordinary VS Code compatibility.<br>Phase: P5 • Priority: P1 • Sources: VS-EXT |
| PLATFORM-03 — Activation and lifecycle | Extensions activate lazily from commands, languages, workspaces, views, URIs, or startup events and dispose resources on shutdown. | Velocity integrations must activate only when needed and remain cancellable; agent sessions cannot assume every tool is always loaded. | Follow native activation patterns and add telemetry/performance budgets for built-in Velocity extensions.<br>Phase: P3 • Priority: P1 • Sources: VS-EXT |
| PLATFORM-04 — Language extensions | Grammar, language configuration, snippets, language servers, debuggers, formatters, linters, test adapters, and notebook kernels extend language support. | Inherit the VS Code language ecosystem instead of rebuilding language tooling. Velocity's agents consume these providers through normalized brokers. | Preserve extension API compatibility and test representative marketplace/Open VSX extensions.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| PLATFORM-05 — Views and tree data providers | Extensions contribute view containers, tree views, welcome content, drag/drop, badges, context actions, and visibility conditions. | Third parties can add project-aware Lens companions, evidence indexes, infrastructure trees, or domain explorers without occupying permanent chrome. | Add Velocity context keys and semantic-resource APIs to tree items.<br>Phase: P5 • Priority: P2 • Sources: VS-EXT |
| PLATFORM-06 — Webviews and webview views | Sandboxed HTML surfaces communicate with extensions through message passing, retain state, load local resources, and enforce content-security policy. | Use for specialized visualizations only. Agent tools must access structured services, not scrape webview DOM. | Require strict CSP, trusted-types, resource roots, serialization, and explicit API contracts.<br>Phase: P3 • Priority: P0 • Sources: VS-EXT |
| PLATFORM-07 — Custom editors and notebooks | Extensions can own editors for structured resources and notebook documents while preserving save, undo, backup, diff, and editor lifecycle. | Allow workflow manifests, API collections, database schemas, and experiment notebooks to remain first-class project artifacts. | Expose native custom editor/notebook APIs and checkpoint their underlying resources.<br>Phase: P5 • Priority: P2 • Sources: VS-EXT;VS-NB |
| PLATFORM-08 — Notebook cells and kernels | Notebook documents contain code/markup cells, outputs, metadata, execution order, kernels, variables, and rich renderers. | Agents may run cells, edit notebooks, inspect outputs, and generate evidence, but execution state and hidden metadata must be captured for reproducibility. | Use notebook controller APIs with environment fingerprints, output limits, and cell-level diffs.<br>Phase: P6 • Priority: P1 • Sources: VS-NB |
| PLATFORM-09 — Notebook diff and trust | Notebooks support structural diffing, output handling, renderer trust, and workspace security constraints. | Checkpoint review must distinguish source-cell changes, metadata changes, and generated outputs. Untrusted rich output never executes automatically. | Reuse notebook diff/trust services and add evidence summaries.<br>Phase: P6 • Priority: P1 • Sources: VS-NB;VS-TRUST |
| PLATFORM-10 — Extension marketplaces | Products can integrate an extension gallery, curated recommendations, allowlists, blocklists, signatures, and update channels. | Velocity must decide whether to use Open VSX, a curated registry, direct VSIX installation, or an enterprise mirror; Microsoft's marketplace terms cannot be assumed transferable. | Configure a compatible gallery and product policy; document supported extension tiers and signing requirements.<br>Phase: P7 • Priority: P0 • Sources: VS-EXT;VOSS-02 |
| PLATFORM-11 — Extension compatibility tiers | Extensions differ in reliance on private APIs, product services, native modules, webviews, authentication, remote hosts, and Microsoft endpoints. | Publish tested tiers: built-in, certified, compatible, best effort, unsupported. Compatibility is a release criterion, not marketing language. | Automate extension smoke tests and maintain a compatibility database per Velocity version.<br>Phase: P7 • Priority: P1 • Sources: VS-EXT |
| PLATFORM-12 — Remote SSH | The UI runs locally while a remote server/extension host accesses files, terminals, debuggers, and tools over SSH. | A Project can live on an SSH host while Velocity retains local UI and policy. Coworker worktrees and browsers may run remotely with proxied evidence. | Reuse remote authority, SSH resolver, server install, and extension placement; add agent session identity.<br>Phase: P6 • Priority: P1 • Sources: VS-REMOTE;VOSS-01 |
| PLATFORM-13 — WSL integration | Windows users can work in Linux distributions with remote files, terminals, extensions, and debugging while retaining desktop UI. | Support WSL as a normal execution authority, including per-distro worktrees and browser/port routing. | Retain upstream WSL remote extension compatibility and add Velocity environment metadata.<br>Phase: P6 • Priority: P2 • Sources: VS-REMOTE |
| PLATFORM-14 — Dev Containers | A repository-defined container provides reproducible tools, runtimes, extensions, mounts, ports, users, and lifecycle commands. | Prefer devcontainer-defined environments for agents when available. Candidate worktrees can mount or clone into isolated containers. | Integrate Dev Containers and treat container build/start logs as environment evidence.<br>Phase: P6 • Priority: P0 • Sources: VS-REMOTE |
| PLATFORM-15 — GitHub Codespaces and remote cloud workspaces | Cloud-hosted development environments expose repository, compute, ports, secrets, and browser access through remote services. | Support as one cloud execution backend without coupling core Velocity to a single provider. | Build a RemoteExecutionProvider interface over existing remote authorities and provider APIs.<br>Phase: P7 • Priority: P2 • Sources: VS-REMOTE |
| PLATFORM-16 — Tunnels and VS Code Server | Secure tunnels connect a client to a remote VS Code Server without direct inbound SSH, preserving workbench capabilities. | Use tunnels for user-owned machines, ephemeral agent runners, and handoff while enforcing project and identity isolation. | Reuse dev-tunnel dependencies and server protocol; add session policy and expiry.<br>Phase: P7 • Priority: P1 • Sources: VOSS-01;VS-REMOTE |
| PLATFORM-17 — Web client | VS Code for the Web runs a browser workbench with web extensions, virtual file systems, limited native execution, and remote backends. | Offer a review/control client that can inspect artifacts and direct agents without requiring full local execution. Heavy tools run remotely. | Compile web workbench and separate control-plane capabilities from native execution-plane services.<br>Phase: P7 • Priority: P2 • Sources: VS-REMOTE |
| PLATFORM-18 — Settings and policy management | Organizations can lock or recommend settings, control extensions, telemetry, updates, AI features, network domains, and browser/agent tools. | Velocity enterprise policy must govern providers, models, network, commands, secrets, deployments, data retention, plugins, browser sharing, and approvals. | Map Velocity policies into native policy services and a signed organization policy bundle.<br>Phase: P7 • Priority: P0 • Sources: VS-AGENTS;VS-BROWSER |
| PLATFORM-19 — Authentication providers | Extensions and product services integrate account sessions, scopes, sign-in UI, secret storage, enterprise hosts, and trusted extension access. | Use brokered identity for Git hosting, cloud providers, design tools, databases, and model vendors; agents receive scoped capabilities, never raw long-lived tokens. | Build on authentication/secret-storage APIs plus an execution-time credential broker.<br>Phase: P5 • Priority: P0 • Sources: VS-EXT;VOSS-02 |
| PLATFORM-20 — Secret storage | Encrypted OS-backed storage exposes secrets to authorized extension contexts without writing them into settings or repositories. | Separate user credentials, project service accounts, agent ephemeral tokens, and production break-glass secrets. Access is time- and scope-bound. | Use native secret storage plus a policy/lease layer and redacted audit records.<br>Phase: P5 • Priority: P0 • Sources: VS-EXT |
| PLATFORM-21 — Telemetry and diagnostics | Core and extensions emit usage/error/performance telemetry under product settings and enterprise policy; local logs support troubleshooting. | Velocity needs transparent, minimizable telemetry with separate product analytics, local agent traces, and customer project evidence. Project content must never become product telemetry by default. | Define event schemas, consent, sampling, retention, local export, and enterprise disablement.<br>Phase: P6 • Priority: P0 • Sources: VS-EXT |
| PLATFORM-22 — Update channels | Stable/insider channels, differential updates, rollback, extension updates, policy controls, and release notes manage distribution. | Because Velocity will track upstream Code-OSS and ship agent/browser capabilities, maintain Stable, Preview, and nightly channels with rollback and migration checks. | Build automated upstream rebase, signed artifacts, staged rollout, crash rollback, and compatibility gates.<br>Phase: P7 • Priority: P0 • Sources: VOSS-01 |
| PLATFORM-23 — Accessibility APIs | Extensions and workbench parts use accessible labels, keyboard navigation, high contrast, screen-reader optimized views, accessible diff, and signals. | Require every Velocity feature to pass keyboard-only, screen-reader, high-contrast, zoom, and reduced-motion acceptance gates; spatial collaboration gets list/timeline equivalents. | Adopt upstream accessibility services and add automated Playwright/axe plus manual assistive-technology testing.<br>Phase: P2 • Priority: P0 • Sources: VS-UI;VOSS-03 |
| PLATFORM-24 — Localization | Product strings, commands, menus, walkthroughs, date/number formats, and extension messages can be localized. | Design mission, decision, evidence, and coworker APIs for localization from the start; do not store localized labels as stable IDs. | Use native localization tooling and message catalogs; keep model-generated text language-aware but not relied upon for state.<br>Phase: P7 • Priority: P2 • Sources: VS-EXT |
| PLATFORM-25 — Enterprise audit export | Organizations need immutable records of agent actions, approvals, code changes, network/tool access, deployments, and policy decisions. | Provide project-controlled audit exports without exposing hidden model reasoning. Store inputs, outputs, tool calls, evidence, decisions, identities, and revisions. | Build append-only signed audit events, retention controls, SIEM export, and privacy review.<br>Phase: P7 • Priority: P0 • Sources: VS-AGENTS |
| PLATFORM-26 — Upstream contribution strategy | A long-lived Code-OSS fork must continuously absorb upstream security, terminal, editor, browser, accessibility, and extension changes. | Keep Velocity changes modular and upstreamable where possible; avoid copying core services into a separate app that immediately diverges. | Maintain a patch queue, ownership map, automated merge cadence, upstream tests, and contribution policy.<br>Phase: P0 • Priority: P0 • Sources: VOSS-01;VOSS-04 |
