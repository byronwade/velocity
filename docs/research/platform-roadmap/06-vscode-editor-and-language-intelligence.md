# 11. VS Code Editor and Language Intelligence

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

The editor is a platform, not a text box. Native text models, language providers, diff/merge, diagnostics, code actions, semantic navigation, and decorations are foundational both for human takeover and for safe agent tools.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| VS-ED-01 — Text buffer and model lifecycle | Files are represented by text models that can be opened in multiple views, edited incrementally, reverted, saved, hot-exited, and restored while preserving undo stacks. | Replace the standalone CodeMirror document model with native text models so every editor, diff, language server, search result, SCM view, and agent patch operates on one authoritative buffer. | Use ITextModel/model services, working-copy services, file services, and native dirty-state semantics.<br>Phase: P1 • Priority: P0 • Sources: VS-EDITOR;VOSS-01 |
| VS-ED-02 — Multi-cursor editing | Users place multiple cursors, add next occurrences, select all matches, and edit columns or disjoint regions simultaneously. | Preserve this as an expert takeover capability. Agent-produced changes should appear as reviewable edits, not interfere with human cursor state. | Use the native editor unchanged; agent patches apply through workspace edits or isolated worktrees.<br>Phase: P1 • Priority: P1 • Sources: VS-EDITOR |
| VS-ED-03 — Selection, column mode, and smart expand | Selections can grow by syntax, operate in columns, select lines, and move/copy blocks while respecting indentation. | No Velocity-specific rewrite is needed; expose commands in Full IDE profile and ensure keybindings remain familiar. | Inherit core editor commands and keymaps from Code-OSS.<br>Phase: P1 • Priority: P1 • Sources: VS-EDITOR |
| VS-ED-04 — Undo, redo, and edit history | The editor maintains local operation history, supports undo stops, and integrates with file-level timeline and SCM history. | Use editor undo for local human actions and checkpoints/Git commits for agent-scale reversibility. Never conflate an agent rollback with Ctrl+Z. | Define transaction boundaries around workspace edits and create Git-backed checkpoints for accepted agent work.<br>Phase: P2 • Priority: P0 • Sources: VS-EDITOR;VEL-02 |
| VS-ED-05 — Find and replace | Search within a file supports regular expressions, case/word controls, selection scope, replacement previews, preserve-case options, and navigation. | Retain native behavior and expose a machine-readable search/replace tool to agents with preview and affected-range evidence. | Reuse editor find controller for humans; agent tool uses search service and WorkspaceEdit with diff preview.<br>Phase: P1 • Priority: P1 • Sources: VS-EDITOR |
| VS-ED-06 — Code folding | Language-aware folding collapses regions, imports, comments, and custom markers, with commands for levels and recursion. | Allow reviewers to collapse unchanged regions and focus on agent edits or failing evidence. Use folding state as presentation only. | Inherit folding providers and add a 'focus changed regions' review command.<br>Phase: P3 • Priority: P2 • Sources: VS-EDITOR |
| VS-ED-07 — Bracket matching and guides | Matching brackets, pair colorization, indentation guides, active bracket guides, and auto-closing improve structural comprehension. | Use standard features. Agent-generated syntax errors should surface through diagnostics and checkpoint validation, not hidden UI. | Inherit editor configuration and theme tokens.<br>Phase: P1 • Priority: P2 • Sources: VS-EDITOR |
| VS-ED-08 — Auto indentation and formatting | Language configuration and formatters control indentation, on-type formatting, on-save formatting, and document/range formatting. | Run formatting as a deterministic post-edit step per work order. Record formatter, version, and files changed in evidence. | Use format providers, save participants, and tasks; pin formatter settings in repository configuration.<br>Phase: P2 • Priority: P0 • Sources: VS-EXT |
| VS-ED-09 — Code actions and quick fixes | Diagnostics can provide fix actions, source actions, organize-imports operations, and refactors at a cursor or range. | Expose safe, deterministic code actions as agent tools. Batch actions require scope, preview, and verification before checkpointing. | Invoke CodeActionProvider results through the command/workspace edit APIs, classify by risk, and log provenance.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT |
| VS-ED-10 — Rename symbol | Language servers compute project-wide symbol renames with conflict validation and preview. | Agents should use semantic rename rather than text replacement whenever available, and checkpoint evidence must list affected references. | Wrap rename providers as a typed tool with preview, cancellation, and path allowlists.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT |
| VS-ED-11 — Go to definition/declaration/type/implementation | Language providers resolve semantic destinations across files and dependencies. | Use these relations to enrich Velocity's project graph, coworker scope selection, impact analysis, and evidence trails. | Consume language-feature providers and index stable symbol IDs in the semantic graph.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT;VEL-02 |
| VS-ED-12 — Find all references | Reference providers enumerate reads, writes, calls, and implementations across the workspace. | Before broad agent edits, compute affected call sites and attach the reference set to blast-radius evidence. | Use reference providers plus ripgrep fallback; cache by model/version and invalidate incrementally.<br>Phase: P3 • Priority: P1 • Sources: VS-EXT |
| VS-ED-13 — Peek views | Definitions, references, implementations, and changes can appear inline without leaving the current file. | Use peek-style reviewers for source evidence and coworker handoffs, keeping the artifact visible while showing related facts. | Reuse peek widgets and reference controllers; contribute commands for checkpoint context.<br>Phase: P3 • Priority: P2 • Sources: VS-EDITOR |
| VS-ED-14 — Outline and symbol navigation | Document symbols create hierarchical outlines, breadcrumbs, sticky scroll, and quick navigation. | Connect file symbols to routes, components, tests, schemas, and reservations so semantic scopes can be selected without raw paths. | Index document symbols into the project graph and expose a unified semantic picker.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT;VEL-02 |
| VS-ED-15 — Workspace symbols | Language servers provide fuzzy-searchable symbols across the workspace. | Use workspace-symbol search as a direct mission targeting mechanism: 'assign this component/service/schema' rather than asking for file paths. | Create a mission scope picker backed by workspace-symbol providers and graph entities.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT |
| VS-ED-16 — Hover information | Providers show types, signatures, documentation, diagnostics, and links near a symbol. | Add compact agent ownership, reservation, last checkpoint, and change-risk metadata as a separate hover participant, without overwhelming language hovers. | Contribute hover content gated by Velocity context and a user setting.<br>Phase: P3 • Priority: P2 • Sources: VS-EXT |
| VS-ED-17 — Signature help | Parameter information appears while calling functions and updates as arguments change. | Preserve native behavior for takeover editing; agents receive the same type information through language tools rather than screen scraping. | Expose language-server requests through an agent language-intelligence service.<br>Phase: P2 • Priority: P2 • Sources: VS-EXT |
| VS-ED-18 — Completion and IntelliSense | Completion providers combine language servers, snippets, words, paths, imports, APIs, and AI suggestions with ranking and filtering. | Keep deterministic IntelliSense independent of generative AI. Coworker inline suggestions must be optional, attributed, cancellable, and separated from autonomous work orders. | Use native completion providers plus a provider-neutral AI completion adapter; apply enterprise policy and telemetry controls.<br>Phase: P4 • Priority: P1 • Sources: VS-EXT;VS-AGENTS |
| VS-ED-19 — Inline suggestions and next edits | Ghost text and predicted next edits appear in context and can be accepted partially or fully. | Offer this only in human takeover mode. Autonomous coworkers should make isolated patches with evidence rather than constantly injecting ghost text. | Integrate through inline completion APIs; disable in review-only or high-assurance profiles.<br>Phase: P4 • Priority: P2 • Sources: VS-AGENTS |
| VS-ED-20 — Snippets | Parameterized templates support tab stops, choices, variables, transformations, language scopes, and project/user distribution. | Use snippets for deterministic scaffolding and human ergonomics. Agents may invoke repository-owned templates but must report provenance. | Reuse snippet service and package Velocity templates as repository assets or extensions.<br>Phase: P3 • Priority: P2 • Sources: VS-EXT |
| VS-ED-21 — Semantic highlighting | Language servers classify tokens semantically beyond grammar syntax, enabling accurate styling for types, members, parameters, and modifiers. | Preserve as core editor fidelity and use token classifications to improve semantic selections and change explanations. | Inherit semantic token providers and theme support.<br>Phase: P1 • Priority: P2 • Sources: VS-EXT |
| VS-ED-22 — Inlay hints | Providers display inferred types, parameter names, references, or other contextual annotations inline. | Velocity can add optional ownership, candidate-change, and test-coverage hints, but these should be off by default and available in review profiles. | Contribute inlay hints with strict density and performance budgets.<br>Phase: P4 • Priority: P3 • Sources: VS-EXT |
| VS-ED-23 — CodeLens | Inline actionable annotations show references, tests, authorship, run/debug actions, and provider-specific commands. | Add mission, owner, checkpoint, coverage, and deployment links above relevant symbols only when they help a current review or conflict. | Contribute CodeLens providers keyed to semantic graph and active work context.<br>Phase: P3 • Priority: P2 • Sources: VS-EXT |
| VS-ED-24 — Sticky scroll | Containing scopes remain visible at the top while scrolling through nested code. | Preserve native behavior; optionally include active reservation/coworker in a subtle adjacent badge. | Inherit core feature; avoid modifying sticky-scroll rendering unless necessary.<br>Phase: P1 • Priority: P3 • Sources: VS-EDITOR |
| VS-ED-25 — Diff editor | Side-by-side or inline diffs support navigation, ignore-whitespace, move detection, revert hunks, accessible diff review, and configurable algorithms. | Make the native diff/multi-diff editor the canonical checkpoint source review. Stable/Candidate, branch/main, and pre/post-format comparisons should all resolve to real resources. | Create checkpoint editor inputs that open native diff and multi-diff models with evidence-linked metadata.<br>Phase: P2 • Priority: P0 • Sources: VS-SCM;VEL-02 |
| VS-ED-26 — Merge editor | Three-way merge presents current, incoming, base, result, conflict actions, and completion checks. | Use it for human conflict resolution when agents overlap or upstream changes invalidate a candidate. Decision Sheets should link directly to unresolved merge regions. | Reuse merge editor and publish conflict events into the orchestration service.<br>Phase: P3 • Priority: P0 • Sources: VS-SCM |
| VS-ED-27 — Readonly and virtual documents | Providers can expose generated, remote, historical, or synthesized content through custom URI schemes and readonly editors. | Represent agent plans, evidence manifests, API responses, logs, prompts, compiled output, and historical checkpoint files as virtual documents when they are not source-of-truth files. | Implement `velocity:` text content providers with clear provenance and export actions.<br>Phase: P3 • Priority: P1 • Sources: VS-EXT |
| VS-ED-28 — Custom editors | Structured resources can have specialized editors with bidirectional synchronization to an underlying document. | Use for design-system manifests, mission specifications, checkpoint bundles, workflow graphs, and structured configuration—but retain a text representation for portability and Git. | Implement custom text editors with schema validation and a source toggle.<br>Phase: P4 • Priority: P2 • Sources: VS-EXT |
| VS-ED-29 — Editor decorations | Extensions and core services can add ranges, glyphs, inline classes, overview markers, and hover information without changing text. | Render current agent edits, reservations, review state, diagnostics, and evidence anchors as decorations. Avoid persistent decorative noise outside active context. | Centralize decoration ownership, z-order, themes, and disposal in a Velocity editor annotation service.<br>Phase: P2 • Priority: P1 • Sources: VS-EXT |
| VS-ED-30 — Language configuration | Bracket pairs, comments, indentation, folding, word patterns, and auto-closing rules can be contributed per language. | Inherit all built-in language support and extension contributions. Do not hard-code language behavior in Velocity. | Use extension host and language configuration registry unchanged.<br>Phase: P1 • Priority: P1 • Sources: VS-EXT |
| VS-ED-31 — Language Server Protocol integration | Language extensions run out of process and provide diagnostics, completions, navigation, refactors, symbols, and semantic data. | Treat language servers as the primary semantic tool plane for agents. Agent context should query symbols and diagnostics before reading entire repositories. | Build a broker that exposes selected language features with cancellation, quotas, caching, and provenance.<br>Phase: P3 • Priority: P0 • Sources: VS-EXT;VS-AGENTS |
| VS-ED-32 — Tree-sitter and structural parsing | Incremental parsers can provide syntax trees where language servers are absent or too expensive. | Use structural parsing for project graph extraction, chunking, safe transformations, and fallback impact analysis. | Leverage the Code-OSS tree-sitter infrastructure and repository parsers behind a normalized syntax service.<br>Phase: P3 • Priority: P1 • Sources: VOSS-01 |
| VS-ED-33 — Formatting preview and save participants | Save can trigger whitespace trimming, final-newline insertion, imports, formatting, and other participants. | Agent checkpoints must distinguish semantic edits from automatic save transforms and avoid infinite edit/format loops. | Run save participants in a controlled transaction and report secondary changes separately.<br>Phase: P2 • Priority: P1 • Sources: VS-SETTINGS |
