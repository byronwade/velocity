# 14. Integrated Terminal and Shell Platform

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Terminal parity is a hard dependency, not a visual feature. Velocity should reuse the fork's native PTY/xterm stack and add work-order identity, policy, isolation, evidence, credential boundaries, budgets, and reproducibility.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| TERM-01 — Native PTY process | A terminal launches a real process through a pseudoterminal, streams bytes bidirectionally, resizes with the viewport, reports exit state, and supports platform-specific shells. | Replace Velocity's virtual interpreter with the Code-OSS `node-pty`/terminal process stack. The virtual shell may survive only as a deterministic test fixture or browser-demo fallback. | Adopt TerminalInstance, TerminalProcessManager, xterm, remote terminal backends, and native process permissions from Velocity-IDE.<br>Phase: P2 • Priority: P0 • Sources: VOSS-01;VOSS-03;VEL-04 |
| TERM-02 — xterm rendering | The terminal uses xterm.js for ANSI/VT parsing, cursor state, selection, alternate buffers, scrollback, links, Unicode, accessibility, GPU rendering, and addons. | Use native xterm unchanged and theme it through workbench tokens. Do not maintain a div-per-line transcript terminal. | Retain xterm and its clipboard, image, search, serialize, Unicode, ligature, WebGL, and progress addons already present in the fork.<br>Phase: P2 • Priority: P0 • Sources: VOSS-01;VOSS-03 |
| TERM-03 — Shell profile discovery | The platform discovers bash, zsh, PowerShell, Command Prompt, fish, Git Bash, Python, WSL distributions, and custom shells, then lets users select defaults. | Show detected profiles and let projects/agents choose approved profiles. Never label a simulated interpreter as zsh or PowerShell. | Use terminal profile resolver services with policy-controlled allowlists and per-platform defaults.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-P;VEL-05 |
| TERM-04 — Custom profiles | Profiles can define executable paths, arguments, environment variables, icons, colors, and source profiles; profiles may be contributed by extensions. | Provide named profiles such as Human Shell, Build, Test, Database, Production Readonly, Coworker Sandbox, and Incident Response. | Extend profile configuration with a Velocity execution policy ID and credential scope.<br>Phase: P3 • Priority: P1 • Sources: VS-TERM-P |
| TERM-05 — Terminal tabs | Multiple terminal instances appear as named tabs with icons, colors, status indicators, process names, exit states, and context actions. | Separate human terminals from coworker terminals by scope and identity, but keep them in the native terminal UI with clear ownership and work-order links. | Add terminal decorations/statuses and a filter/group view, not a separate agent terminal implementation.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-B |
| TERM-06 — Terminal groups and splits | Terminals can be split horizontally/vertically, grouped, moved, reordered, resized, and navigated with keyboard commands. | Use groups for service topologies, paired server/client processes, test watchers, and agent sub-tasks. Follow Mode may reveal a coworker's group without taking ownership. | Reuse terminal group services and serialize group topology per workstream.<br>Phase: P2 • Priority: P1 • Sources: VS-TERM-B |
| TERM-07 — Terminal editors | A terminal can live in the editor area, move between panel and editor, and participate in editor groups and auxiliary windows. | Allow long-running servers, remote shells, and production logs to become durable editors while the bottom panel remains compact. | Use terminal editor inputs and serializers; keep process lifecycle independent of view placement.<br>Phase: P3 • Priority: P1 • Sources: VS-TERM-B;VOSS-03 |
| TERM-08 — Terminal persistence | Terminal buffers and processes can persist across window reloads; sessions may reconnect or revive depending on platform/configuration. | Persist human and approved service terminals, but default autonomous agent terminals to mission/work-order lifetimes with explicit revival rules. | Use persistent/reconnection properties and serialize only non-secret metadata; tie agent cleanup to work-order state.<br>Phase: P3 • Priority: P0 • Sources: VS-TERM-A;VOSS-03 |
| TERM-09 — Detach and attach | Terminal processes can be detached from one client and attached to another or restored through remote/reconnection services. | Support handing a live process from an agent session to a human, moving between windows, or reconnecting to a cloud/SSH runtime without restarting. | Expose a controlled handoff command with ownership, audit, and input-permission transitions.<br>Phase: P5 • Priority: P1 • Sources: VS-TERM-A |
| TERM-10 — Shell integration injection | VS Code injects shell scripts for supported shells to identify command boundaries, current working directory, exit codes, environment changes, and prompt state. | This is essential for evidence: Velocity should know which command ran, where, whether it succeeded, what files/processes it affected, and which work order invoked it. | Retain shell integration for bash/zsh/fish/PowerShell and treat missing integration as degraded evidence, not silent success.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-S;VOSS-03 |
| TERM-11 — Command decorations | Detected commands receive gutter decorations and context actions for rerun, copy command/output, explanation, and navigation. | Decorate commands with coworker, work order, approval, risk, and evidence status. A user can jump from a checkpoint to the exact command/output block. | Contribute terminal command actions and metadata through shell integration capabilities.<br>Phase: P3 • Priority: P1 • Sources: VS-TERM-S |
| TERM-12 — Command history and navigation | Users navigate prior commands, recent commands, shell history, and command blocks independently of raw scrollback. | Provide a per-project command ledger that can filter by human/coworker, mission, exit code, tool, and risk while respecting secret redaction. | Index shell-integration command events into the event store; retain native history UX.<br>Phase: P3 • Priority: P1 • Sources: VS-TERM-S |
| TERM-13 — Current working directory tracking | Shell integration reports cwd changes and enables opening files, splitting in the same directory, and resolving relative links accurately. | Bind each work-order terminal to its worktree root and flag attempts to escape allowed roots. Cwd becomes part of evidence and command policy. | Combine terminal capability data with a sandbox path policy and worktree identity.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-S |
| TERM-14 — Environment variable collections | Extensions can declaratively add, prepend, append, or replace variables in terminal environments and show environment-change indicators. | Build reproducible agent environments from project, provider, secret broker, and mission layers. Do not inject secrets into broad persistent shells. | Use environment collections plus ephemeral credential handles; display provenance and expiration.<br>Phase: P3 • Priority: P0 • Sources: VS-TERM-A;VOSS-03 |
| TERM-15 — Environment changes and relaunch | When profile or environment configuration changes, terminals can show stale-environment status and offer relaunch. | Mark agent runs non-reproducible if environment changes mid-work. New work orders should use a fresh environment snapshot. | Hash environment manifests and attach them to run evidence; require relaunch for policy-critical changes.<br>Phase: P3 • Priority: P1 • Sources: VS-TERM-A |
| TERM-16 — Working-directory launch policies | New terminals can start at the workspace root, active file directory, selected folder, inherited terminal cwd, or a configured path. | Default human shells to project root and coworker shells to isolated worktrees. Database/deployment shells should start in approved tool contexts. | Resolve launch cwd through a Velocity terminal factory with explicit scope metadata.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-B |
| TERM-17 — Terminal links | The terminal detects web URLs, local/remote files, folders, line/column references, ports, and extension-provided link patterns. | Turn localhost links into session-scoped browser tabs and file references into the exact worktree resource. Production URLs require risk-aware open behavior. | Use terminal link providers that preserve remote authority, agent session, and environment identity.<br>Phase: P2 • Priority: P1 • Sources: VS-TERM-B;VS-BROWSER |
| TERM-18 — Port and localhost integration | Detected server output can reveal or forward ports, open browsers, and connect remote services to local clients. | Associate every port with process, worktree, environment, owner, visibility, and lifecycle; never assume `localhost:3000` is globally unique. | Integrate terminal links, Ports view, tunneling service, and browser session routing.<br>Phase: P3 • Priority: P0 • Sources: VS-TERM-B;VS-REMOTE |
| TERM-19 — Selection and clipboard | Terminal selection supports line, word, column, keyboard selection, copy-on-select options, paste, bracketed paste, and multiline paste warnings. | Preserve expert controls. Clipboard writes/reads by agents require explicit permission and should not silently enter shared human clipboard state. | Retain core behavior; add agent-tool clipboard denial by default and audit user-approved transfers.<br>Phase: P2 • Priority: P1 • Sources: VS-TERM-B |
| TERM-20 — Search in scrollback | Find supports regex, case, whole word, incremental results, and navigation through terminal output. | Make terminal output searchable and link search results to evidence. Large outputs should be persisted and indexed without overloading model context. | Reuse xterm search addon and spool structured output to bounded log artifacts.<br>Phase: P2 • Priority: P1 • Sources: VS-TERM-B;VOSS-01 |
| TERM-21 — Scrollback and buffer serialization | Terminals keep configurable scrollback, alternate-screen state, and serialized buffers for restoration. | Store enough output for review while applying per-project retention and secret redaction. Never treat raw scrollback as the only audit record. | Use xterm serialize/headless addons plus structured command records and output artifact storage.<br>Phase: P3 • Priority: P0 • Sources: VS-TERM-A;VOSS-01 |
| TERM-22 — Unicode, emoji, and grapheme handling | Unicode versions, wide characters, combining marks, emoji, and ambiguous-width behavior are handled by xterm and configuration. | Inherit the mature renderer and test across platforms; do not custom-render agent output in a simplified transcript. | Retain Unicode addon and upstream terminal tests.<br>Phase: P2 • Priority: P2 • Sources: VS-TERM-A;VOSS-01 |
| TERM-23 — Images and rich terminal protocols | Supported terminals can display inline images and progress information through terminal protocols/addons. | Capture rich output as evidence when useful, but provide accessible text alternatives and avoid sending opaque images to models without consent. | Use image/progress addons with artifact extraction and accessibility summaries.<br>Phase: P5 • Priority: P3 • Sources: VS-TERM-A;VOSS-01 |
| TERM-24 — Ligatures and GPU rendering | Optional ligatures and WebGL acceleration improve typography and performance, with fallback paths. | Inherit settings and performance telemetry; Velocity should not force visual effects that compromise compatibility. | Retain xterm addons and upstream fallback logic.<br>Phase: P2 • Priority: P3 • Sources: VOSS-01 |
| TERM-25 — Fixed dimensions | Users can set fixed terminal columns/rows for predictable wrapping and remote applications. | Useful for reproducing CI or snapshot output; expose as an advanced command and evidence attribute. | Reuse terminal dimensions override and include cols/rows in run manifests.<br>Phase: P4 • Priority: P3 • Sources: VS-TERM-B;VOSS-03 |
| TERM-26 — Keybinding arbitration | The terminal decides whether keystrokes go to the shell or trigger workbench commands, with commands-to-skip-shell and chord behavior. | Preserve native arbitration. Velocity shortcuts must use context keys and must not steal shell input, especially inside full-screen TUIs. | Register Velocity keybindings conservatively and test terminal focus contexts.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-A;VS-KEYS |
| TERM-27 — Signals and process control | Keyboard gestures and commands send interrupts, EOF, suspend, terminate, and other platform-specific signals; process exit reasons are tracked. | Agents need explicit cancellation semantics: soft interrupt, timeout, graceful stop, hard kill, and orphan cleanup, each recorded in the run trace. | Use terminal/process manager APIs and a work-order cancellation state machine.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-A;VOSS-03 |
| TERM-28 — Paste and unsafe input confirmation | Multiline or suspicious pasted text can trigger confirmation; shell integration and settings influence warnings. | Apply equivalent or stricter gates to agent-issued commands, especially when commands contain destructive flags, network exfiltration, privilege elevation, or encoded payloads. | Build a command risk classifier and policy decision point before PTY input.<br>Phase: P2 • Priority: P0 • Sources: VS-TERM-B;VS-AGENTS |
| TERM-29 — Auto replies | Configured pattern-response rules can automatically answer known terminal prompts. | For autonomous runs, use narrowly scoped prompt handlers with expiration and audit; never create global password or approval auto-replies. | Extend auto replies with work-order scope, allowed patterns, non-secret values, and policy review.<br>Phase: P4 • Priority: P1 • Sources: VS-TERM-A |
| TERM-30 — Tasks integration | Tasks launch in terminals with presentation, reveal, panel, group, reuse, focus, clear, close, and problem-matcher behavior. | Agents should prefer repository tasks over arbitrary shell commands when equivalent because tasks encode canonical intent and diagnostics. | Expose task discovery/execution as the first tool tier; fall back to shell only when required.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS;VS-TERM-B |
| TERM-31 — Problem matchers | Task/terminal output is parsed into file-linked diagnostics with severity, owner, watching state, and background lifecycle. | All build, lint, test, and deploy commands should publish structured Problems and evidence rather than relying on natural-language log interpretation. | Reuse and extend problem matchers; let agents propose matchers but require repository review.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS |
| TERM-32 — Command execution policies | Native terminals execute with the user's permissions; agent systems add sandboxing, approvals, allowlists, and network restrictions. | Create four run modes: readonly inspection, workspace-write sandbox, network-limited autonomous, and explicit full-access. Policy is evaluated per command and tool call. | Place a Velocity policy broker before process launch/input, integrated with workspace trust, enterprise policy, and mission envelopes.<br>Phase: P3 • Priority: P0 • Sources: VS-AGENTS;CUR-TERM |
| TERM-33 — Per-agent terminal namespace | Parallel agents need separate process trees, cwd, environment, output, ports, and cleanup even when visible in one workbench. | Each coworker/work order receives isolated terminals bound to its worktree and browser profile; shared terminals are opt-in handoff objects. | Create an AgentExecutionContext propagated through terminal creation, process metadata, ports, evidence, and cancellation.<br>Phase: P4 • Priority: P0 • Sources: CUR-AWIN;CUR-WT;VS-AGENTSW |
| TERM-34 — Terminal-to-agent context | Selected output, command blocks, failures, links, and diagnostics can be attached to an agent without copying the entire terminal. | Let users or orchestrators share bounded command records, redacted output, exit code, cwd, and environment hash. Agents should request more only when needed. | Implement a terminal context provider with redaction, token budgeting, provenance, and share confirmation for human terminals.<br>Phase: P4 • Priority: P0 • Sources: VS-AGENTS;VS-TERM-S |
| TERM-35 — Agent command planning and preview | Before executing, an agent can explain a command, show why it is needed, identify expected effects, and request approval based on risk. | Make this mandatory for destructive, privileged, production, secret-bearing, or out-of-scope commands; low-risk commands may auto-run under policy. | Add plan records, normalized command AST where possible, policy decision logs, and approve-once/approve-pattern choices.<br>Phase: P4 • Priority: P0 • Sources: VS-AGENTS;CUR-TERM |
| TERM-36 — Run trace and reproducibility | A reliable run record includes executable/profile, arguments or typed command, cwd, environment fingerprint, start/end times, exit code, output references, files changed, network/port activity, and initiator. | Attach terminal traces to every checkpoint so a reviewer can understand and, where safe, reproduce verification without reading chat. | Build an immutable command-run entity and correlation IDs across terminal, tasks, browser, tests, and SCM.<br>Phase: P3 • Priority: P0 • Sources: VEL-02 |
| TERM-37 — Secret redaction | Terminal output may contain API keys, tokens, passwords, cookies, connection strings, or customer data. | Redact known secret values and high-confidence patterns before persistence or model context while preserving a sealed local raw log under policy if required. | Integrate secret broker fingerprints, streaming redaction, user review, retention, and audit controls.<br>Phase: P4 • Priority: P0 • Sources: VS-AGENTS;CUR-TERM |
| TERM-38 — Terminal acceptance gate | Production readiness requires real shell behavior, process control, profile parity, splits/tabs, persistence, shell integration, tasks, diagnostics, links/ports, accessibility, remote operation, and agent isolation. | Declare the current virtual shell non-production until this gate passes on macOS, Windows, Linux, WSL/SSH/container, and reconnect scenarios. | Adopt upstream terminal test suites plus Velocity end-to-end tests for worktree isolation, evidence capture, approvals, and cleanup.<br>Phase: P2 • Priority: P0 • Sources: VOSS-03;VS-TERM-B;VS-TERM-S;VS-TERM-A |
