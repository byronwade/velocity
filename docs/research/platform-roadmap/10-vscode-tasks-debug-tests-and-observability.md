# 15. Tasks, Debugging, Testing, Ports, and Observability

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

The complete development loop must flow through repository-native tasks, debugger/test adapters, problem matchers, ports, and structured results. These services generate the evidence that makes autonomous work reviewable.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| DEVLOOP-01 — Task auto-detection | Extensions infer build, test, lint, package, container, and script tasks from project files without requiring manual configuration. | Ingest detected tasks into the project capability graph and prefer them for agent execution because they represent established workflows. | Use task providers and cache task identities per workspace revision.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS |
| DEVLOOP-02 — tasks.json configuration | Repositories define commands, shell/process execution, options, environment, cwd, presentation, groups, problem matchers, dependencies, inputs, and OS overrides. | Treat repository tasks as versioned product configuration. Agents may propose additions but changes require normal code review and verification. | Use native task schema/validation/editor and surface task diffs in checkpoints.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS |
| DEVLOOP-03 — Shell versus process tasks | Tasks can invoke through a shell or launch a process directly, affecting quoting, environment expansion, and security. | Risk classification must understand which execution path is used. Prefer process tasks for deterministic tooling and shell tasks when shell semantics are required. | Inspect resolved task execution and apply policy before launch.<br>Phase: P3 • Priority: P0 • Sources: VS-TASKS |
| DEVLOOP-04 — Task variables and inputs | Commands resolve workspace/file/environment/configuration variables and can prompt for strings, picks, or commands. | Autonomous runs cannot stall on hidden prompts. Inputs must be pre-resolved, explicitly requested, or supplied by approved non-secret automation. | Add an agent task resolver that reports unresolved inputs and requests structured decisions.<br>Phase: P3 • Priority: P1 • Sources: VS-TASKS |
| DEVLOOP-05 — Task compounds and dependencies | Tasks can depend on others sequentially or in parallel and can be grouped as build/test/default tasks. | Represent canonical project workflows as task graphs and map their nodes to evidence, time, cost, and failure ownership. | Use task execution events and dependency metadata; avoid duplicating orchestration for ordinary build graphs.<br>Phase: P3 • Priority: P1 • Sources: VS-TASKS |
| DEVLOOP-06 — Background tasks and watchers | Long-running tasks signal active/ready states through background problem matchers and can be reused for incremental feedback. | Associate watcher processes with worktrees, ports, and missions; readiness must be machine-detected before browser/test steps proceed. | Use background task lifecycle plus process/port correlation.<br>Phase: P3 • Priority: P0 • Sources: VS-TASKS |
| DEVLOOP-07 — Task presentation controls | Tasks specify reveal, focus, panel reuse, grouping, clearing, close behavior, echo, and terminal location. | Keep autonomous output quiet unless a failure, approval, or Follow Mode demands attention. Humans can always open the underlying terminal. | Set agent task presentation defaults and preserve user overrides for human runs.<br>Phase: P3 • Priority: P1 • Sources: VS-TASKS |
| DEVLOOP-08 — Problem matchers | Regex/state machines parse output into diagnostics, including background begin/end patterns and file locations. | Convert task failures into structured Problems and checkpoint evidence; train agents on diagnostics, not raw log guessing. | Use built-in/custom matchers and validate matcher accuracy in project onboarding.<br>Phase: P2 • Priority: P0 • Sources: VS-TASKS |
| DEVLOOP-09 — Debug adapter protocol | Language/runtime-specific debug adapters communicate through a standard protocol for launch, attach, breakpoints, stack, variables, evaluation, memory, disassembly, and stepping. | Expose debugging as a provider-neutral agent tool plane. Coworkers can start or attach, inspect state, and gather evidence while permissions constrain evaluation and side effects. | Reuse DAP services and create a guarded DebugTool facade.<br>Phase: P3 • Priority: P0 • Sources: VS-DEBUG |
| DEVLOOP-10 — Launch configurations | `launch.json` describes type, request, program, cwd, environment, arguments, source maps, pre/post tasks, consoles, and compounds. | Agents should discover and use existing launch configurations first. New configurations are source changes and require review. | Index configurations by stable name and resolve variables through native services.<br>Phase: P3 • Priority: P0 • Sources: VS-DEBUG |
| DEVLOOP-11 — Run and Debug view | A launch picker, configuration controls, variables, watch expressions, call stack, breakpoints, and debug console form the debug workspace. | Velocity can surface a Debug Lens or open native views on demand. The default stage remains the running artifact and current evidence. | Reuse native views and add mission/checkpoint context rather than rebuilding debugging UI.<br>Phase: P3 • Priority: P1 • Sources: VS-DEBUG |
| DEVLOOP-12 — Breakpoints | Line, conditional, hit-count, log, function, data, triggered, and exception breakpoints can be enabled, grouped, and persisted. | Let agents place temporary session-scoped breakpoints and logpoints, while user breakpoints remain untouched. Evidence records breakpoint intent and results. | Tag breakpoint ownership and clean agent breakpoints at session end.<br>Phase: P4 • Priority: P1 • Sources: VS-DEBUG |
| DEVLOOP-13 — Stepping and execution control | Continue, pause, step over/into/out, restart frame, stop, disconnect, and reverse actions depend on adapter capabilities. | Offer bounded agent debugging loops with maximum steps/time and stop conditions to prevent runaway inspection. | Wrap debug actions in work-order budgets and cancellation controls.<br>Phase: P4 • Priority: P0 • Sources: VS-DEBUG |
| DEVLOOP-14 — Variables and data inspection | Scopes, variables, object expansion, data breakpoints, memory, registers, and custom visualizers expose runtime state. | Capture only relevant state snapshots as evidence, with secret/PII redaction and size limits. Never persist entire process memory by default. | Build a selective debug snapshot artifact with policy filters.<br>Phase: P4 • Priority: P0 • Sources: VS-DEBUG |
| DEVLOOP-15 — Watch and evaluation | Expressions can be watched or evaluated in selected stack frames; evaluation may execute code depending on runtime. | Classify evaluation as potentially side-effecting. Agents default to readonly inspection and require approval for expressions that may mutate state. | Add adapter/runtime risk policy and evaluation audit logs.<br>Phase: P4 • Priority: P0 • Sources: VS-DEBUG;VS-AGENTS |
| DEVLOOP-16 — Debug console | A REPL-like console shows adapter output and evaluates expressions in debug context. | Integrate console commands into the same evidence and permission system as terminals, without conflating the two execution environments. | Record debug-console evaluations with frame/session identity and redaction.<br>Phase: P4 • Priority: P1 • Sources: VS-DEBUG |
| DEVLOOP-17 — Multi-target debugging | Compound configurations and multi-session support debug client/server, services, workers, browser/server, and distributed processes. | Attach a mission to several targets and correlate call stacks, logs, ports, and browser evidence across them. | Use compound sessions plus correlation IDs and topology metadata.<br>Phase: P5 • Priority: P1 • Sources: VS-DEBUG |
| DEVLOOP-18 — Browser debugging | JavaScript debugging launches or attaches to browsers, maps source, sets breakpoints, profiles code, and inspects page/runtime state. | Use the integrated browser and `editor-browser` debugging for web work so browser actions, console/network evidence, and source debugging share one session identity. | Adopt the Code-OSS JS debug stack and browser editor integration.<br>Phase: P3 • Priority: P0 • Sources: VS-BROWSER;VS-DEBUG |
| DEVLOOP-19 — Test discovery | Testing extensions publish hierarchical controllers, items, tags, states, locations, and dynamic discovery into Test Explorer. | Index tests as semantic project entities linked to files, symbols, routes, requirements, and checkpoints. | Consume test controller APIs and maintain a test-impact graph.<br>Phase: P3 • Priority: P0 • Sources: VS-TEST |
| DEVLOOP-20 — Run and debug tests | Users run or debug all, suites, files, individual tests, or selections through named test profiles. | Agents select the smallest relevant profile based on impact, then expand to broader verification gates before checkpoint readiness. | Build a test selection policy using changed symbols, dependencies, risk, and project rules.<br>Phase: P3 • Priority: P0 • Sources: VS-TEST |
| DEVLOOP-21 — Test profiles | Extensions provide run, debug, coverage, continuous, and custom profiles with defaults and configuration. | Map mission acceptance criteria to required profiles; a checkpoint records exact controller/profile/test IDs and revisions. | Store profile requirements in project/mission policy and evidence manifests.<br>Phase: P3 • Priority: P0 • Sources: VS-TEST |
| DEVLOOP-22 — Continuous testing | Tests can rerun on changes, show live status, and provide rapid feedback during editing. | Run impacted tests in candidate worktrees without stealing resources from human sessions. Debounce agent edit bursts and cancel superseded runs. | Create per-worktree continuous-test coordinators with concurrency limits.<br>Phase: P4 • Priority: P1 • Sources: VS-TEST |
| DEVLOOP-23 — Test results and messages | Pass/fail/skip states, durations, output, stack traces, expected/actual data, and locations are presented and navigable. | Normalize results into evidence; failures become work items assigned back to the responsible coworker unless policy escalates. | Persist structured test run/result entities and link them to source revisions.<br>Phase: P3 • Priority: P0 • Sources: VS-TEST |
| DEVLOOP-24 — Coverage | Coverage profiles expose file, statement, branch, declaration, and per-test coverage with editor annotations. | Use coverage as evidence and impact guidance, not a vanity target. Show changed-code coverage and regression from Stable. | Compute delta coverage for checkpoint diffs and enforce project-defined floors.<br>Phase: P4 • Priority: P1 • Sources: VS-TEST |
| DEVLOOP-25 — Flaky-test handling | Repeated outcomes and historical durations can distinguish stable failures from intermittent behavior, though frameworks vary. | Track test reliability over time, rerun within bounded policy, and never mark a flaky pass as clean without disclosure. | Add flake statistics, quarantine metadata, and evidence confidence scoring.<br>Phase: P5 • Priority: P1 • Sources: VS-TEST |
| DEVLOOP-26 — Ports view | Forwarded and detected ports expose process, origin, visibility, protocol, label, and open/copy actions, especially in remote contexts. | Create a service registry per worktree/session so browsers and agents route to the correct app and cleanup ports with owners. | Integrate Ports/tunnel services and terminal/browser correlation.<br>Phase: P3 • Priority: P0 • Sources: VS-REMOTE;VS-BROWSER |
| DEVLOOP-27 — Port forwarding and tunnels | Remote ports can be forwarded, labeled, protocol-selected, made private/public under policy, and accessed through secure tunnels. | Treat public exposure as a protected action. Temporary previews default private with expiration and owner. | Use dev tunnels/remote services already present in Velocity-IDE plus approval and expiry policy.<br>Phase: P5 • Priority: P0 • Sources: VOSS-01;VS-REMOTE |
| DEVLOOP-28 — Logs and output correlation | Tasks, debug sessions, tests, language servers, terminals, browsers, extensions, and applications emit separate streams. | Velocity needs one trace index with correlation IDs, time ranges, source revisions, owners, and deep links, while retaining native specialized views. | Implement an event bus and observability index over native output/log/test/debug/browser events.<br>Phase: P3 • Priority: P0 • Sources: VEL-02 |
| DEVLOOP-29 — Performance profiling | Debuggers and extensions can collect CPU profiles, heap snapshots, flame charts, and runtime-specific performance data. | Allow agents to gather profiles under explicit budget, compare Stable/Candidate, identify regressions, and attach summarized evidence. | Integrate profiling adapters/tools with artifact size limits and privacy controls.<br>Phase: P6 • Priority: P2 • Sources: VS-DEBUG |
| DEVLOOP-30 — Test and debug acceptance gate | A production build must discover repository-native tasks, launch/debug configurations, tests, coverage, ports, and diagnostics across local and remote environments. | The vertical slice is incomplete until an agent can reproduce a bug, debug it, edit in isolation, run impacted and required tests, and produce reviewable evidence. | Create cross-language fixtures and end-to-end gates for JavaScript/TypeScript, Python, Rust or Go, browser apps, and remote containers.<br>Phase: P3 • Priority: P0 • Sources: VS-TASKS;VS-DEBUG;VS-TEST |
