# 22. Cursor Customization, Governance, Plugins, and CLI

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Rules, skills, subagents, hooks, MCP, plugins, headless execution, approvals, and security must become inspectable, versioned, permissioned Velocity assets rather than opaque prompt folders.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| CUR-CUST-01 — Project rules | Repository-scoped rule files provide persistent instructions, conventions, architecture constraints, and path-specific guidance. | Store project rules in version control, validate scope/precedence, and make every active rule visible in the work-order context manifest. | Define `.velocity/rules/*.md` plus metadata schema and provider adapters.<br>Phase: P5 • Priority: P0 • Sources: CUR-RULES |
| CUR-CUST-02 — User rules | Personal preferences can apply across projects without modifying repositories. | Keep personal style/ergonomic preferences separate from organization/project requirements and never let them weaken security or acceptance gates. | Use profile/user storage with precedence below policy and above optional defaults.<br>Phase: P5 • Priority: P1 • Sources: CUR-RULES |
| CUR-CUST-03 — Scoped rules | Rules can activate for file patterns, folders, languages, or explicit references. | Extend scoping to semantic components, services, schemas, environments, tools, and mission types while retaining understandable precedence. | Implement rule matcher over URI, language, graph tags, work-order type, and policy context.<br>Phase: P5 • Priority: P0 • Sources: CUR-RULES |
| CUR-CUST-04 — Rule generation and migration | Existing instructions or conventions can be converted into structured rules. | Provide an importer for Cursor rules, VS Code instructions, CLAUDE.md, AGENTS.md, CONTRIBUTING, lint configs, and architecture docs, but require human review of generated rules. | Build instruction discovery, conflict report, and migration assistant.<br>Phase: P5 • Priority: P1 • Sources: CUR-RULES;VEL-03 |
| CUR-CUST-05 — Skills | Reusable skill packages encode procedural workflows, references, scripts, templates, and quality checks. | Model skills as versioned, inspectable capability bundles assigned to coworkers or work orders, with declared tools, inputs, outputs, and trust. | Create Skill manifest, registry, installer, sandbox, and test suite.<br>Phase: P5 • Priority: P0 • Sources: CUR-SKILLS |
| CUR-CUST-06 — Skill discovery | Agents can find applicable skills based on task and project context. | Rank skills by explicit assignment, project declaration, semantic match, capability, version, trust, and prior outcomes; show selected skills in the plan. | Build skill resolver and explainable selection record.<br>Phase: P6 • Priority: P1 • Sources: CUR-SKILLS |
| CUR-CUST-07 — Skill execution | A skill may invoke scripts or tools and return structured artifacts. | Execute through the ToolBroker under the work-order capability envelope; a skill cannot bypass approvals or gain hidden tools. | Compile skill steps to typed tool calls with provenance and cancellation.<br>Phase: P6 • Priority: P0 • Sources: CUR-SKILLS |
| CUR-CUST-08 — Subagent definitions | Specialized subagents have roles, prompts/instructions, tool sets, models, and return expectations. | Translate directly to Velocity Specialists and Work Orders, but keep human-facing identity at the coworker layer. | Define SpecialistTemplate and WorkOrderContract schemas.<br>Phase: P5 • Priority: P0 • Sources: CUR-SUB;VEL-02 |
| CUR-CUST-09 — Hooks | Lifecycle hooks run before or after agent actions, tools, edits, commits, or session events. | Use hooks for policy, formatting, tests, secret scanning, evidence enrichment, notifications, and integrations; hooks must be bounded and visible. | Create hook registry with ordering, timeout, failure mode, and no-bypass policy.<br>Phase: P6 • Priority: P0 • Sources: VS-AGENTS;CUR-PLUG |
| CUR-CUST-10 — MCP server registration | Projects or users configure MCP servers, transports, commands, environment, authentication, and enabled tools. | Provide a registry that displays server origin, publisher, tools, permissions, network access, secrets, and trust status before enabling. | Build MCP management UI and signed configuration schema.<br>Phase: P5 • Priority: P0 • Sources: CUR-MCP |
| CUR-CUST-11 — MCP tool discovery | Agents learn schemas and descriptions from connected MCP servers. | Normalize schemas, classify risk, cap result size, and avoid injecting all tool descriptions into every context unnecessarily. | Implement lazy discovery, caching, policy metadata, and version fingerprints.<br>Phase: P5 • Priority: P0 • Sources: CUR-MCP |
| CUR-CUST-12 — MCP resources and prompts | Servers may expose resources or prompt templates in addition to executable tools. | Represent resources as provenance-rich ResourceRefs and prompts as inspectable templates; neither gains execution authority automatically. | Map through ContextBroker and Template registry.<br>Phase: P6 • Priority: P1 • Sources: CUR-MCP |
| CUR-CUST-13 — MCP authentication | External servers may need OAuth, API keys, local credentials, or enterprise identity. | Use brokered sessions and secret leases; never store tokens in plain MCP config or expose them to models. | Integrate authentication providers, secret storage, and server-specific scopes.<br>Phase: P6 • Priority: P0 • Sources: CUR-MCP |
| CUR-CUST-14 — Plugin packaging | Cursor plugins can bundle tools, rules, commands, or integrations for installation. | Define Velocity plugins as signed packages that may contribute extensions, skills, policies, tools, views, and templates with declared capabilities. | Build package format atop VSIX where possible plus Velocity manifest.<br>Phase: P7 • Priority: P1 • Sources: CUR-PLUG;VS-EXT |
| CUR-CUST-15 — Plugin permissions | Installed plugins may access files, network, terminals, secrets, browser, or project metadata. | Show install-time and runtime capabilities, organization policy, provenance, signatures, updates, and revocation. High-risk capability changes require re-consent. | Create permission manifest and capability broker integration.<br>Phase: P7 • Priority: P0 • Sources: CUR-PLUG |
| CUR-CUST-16 — Plugin marketplace | A catalog supports discovery, installation, updates, ratings, compatibility, and governance. | Start curated and signed; separate Code-OSS extension compatibility from Velocity-certified autonomous tools. | Build registry, certification process, enterprise mirror, and compatibility metadata.<br>Phase: P8 • Priority: P1 • Sources: CUR-PLUG;VS-EXT |
| CUR-CUST-17 — Headless CLI | Cursor can run agents non-interactively for scripts, CI, terminals, or automation. | Ship a `velocity` CLI that creates/continues work orders, runs plans, streams structured events, exports artifacts, and exits with policy-aware status. | Implement CLI over the same local/remote orchestration API, not a separate agent implementation.<br>Phase: P7 • Priority: P0 • Sources: CUR-CLI |
| CUR-CUST-18 — Structured CLI output | Headless runs need JSON or machine-readable events for CI and downstream automation. | Define stable JSON Lines events for plans, tools, changes, evidence, approvals, costs, checkpoints, and final status. | Version the event schema and provide redaction/filtering.<br>Phase: P7 • Priority: P0 • Sources: CUR-CLI |
| CUR-CUST-19 — CI integration | Headless agents can review, fix, test, or generate artifacts inside continuous-integration workflows. | Default CI runs to readonly review or draft patch artifacts; write/push behavior requires explicit bot identity and protected-branch policy. | Provide CI templates, OIDC credentials, ephemeral runners, and checkpoint export.<br>Phase: P7 • Priority: P0 • Sources: CUR-CLI;CUR-AUTO |
| CUR-CUST-20 — Approval agents | Policy-focused agents can assess requests and decide or recommend within configured authority. | Treat them as deterministic-policy augmented reviewers; record evidence, policy version, confidence, and escalation. They cannot override hard policy. | Implement evaluator interface and authority matrix.<br>Phase: P7 • Priority: P0 • Sources: CUR-APPROVAL |
| CUR-CUST-21 — Security model | Cursor documents sandboxing, approvals, network controls, worktree isolation, and caution around untrusted instructions/tools. | Velocity must assume prompt injection can arrive from source files, web pages, terminal output, issues, docs, MCP, and generated artifacts. | Build layered trust labels, content/tool separation, egress controls, and high-risk action approvals.<br>Phase: P4 • Priority: P0 • Sources: CUR-TERM;CUR-BROW;VS-AGENTS |
| CUR-CUST-22 — Prompt-injection defenses | External content can attempt to redirect agents, exfiltrate secrets, or trigger unsafe tools. | Mark untrusted content, prevent it from modifying system/policy instructions, limit tool grants, and require confirmation for content-originated protected actions. | Implement taint/provenance labels in ContextBroker and policy rules.<br>Phase: P5 • Priority: P0 • Sources: VS-AGENTS;CUR-BROW |
| CUR-CUST-23 — Auditability | Organizations need to inspect what agents read, changed, executed, accessed, and approved. | Provide replayable tool/action logs and source/evidence manifests without exposing private hidden reasoning or unnecessary project content. | Create signed event log, export, retention, access controls, and redaction.<br>Phase: P6 • Priority: P0 • Sources: CUR-APPROVAL;VS-AGENTS |
| CUR-CUST-24 — Model/provider governance | Organizations may restrict models, data residency, retention, training use, regions, or fallback behavior. | Resolve provider/model choices through organization policy before coworker preferences; surface effective policy and fallback path. | Implement provider catalog, policy engine, contracts metadata, and run manifest fields.<br>Phase: P6 • Priority: P0 • Sources: CUR-AGENT;VEL-02 |
| CUR-CUST-25 — Customization acceptance gate | A customization is production-ready only when versioned, inspectable, scoped, permissioned, testable, migratable, and portable across providers where promised. | Reject opaque prompt folders or plugins that silently gain tools. Velocity's differentiation depends on trustworthy composition. | Publish schema validators, conformance tests, compatibility reports, and safe-mode disablement.<br>Phase: P6 • Priority: P0 • Sources: CUR-RULES;CUR-SKILLS;CUR-MCP;CUR-PLUG |
