# 25. Figma Design Systems, Variables, Inspection, and Dev Mode

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Figma's components, variables, libraries, Dev Mode, Code Connect, annotations, change comparison, and design checks translate into a source-backed semantic design system and agent-first inspection workflow.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| FIG-DS-01 — Components | Figma components define reusable UI building blocks whose instances inherit structure and properties. | Treat source components as the canonical reusable units. The design canvas visualizes them and lets agents modify component source, tests, stories, and docs together. | Index framework components and map rendered instances to component definitions.<br>Phase: P5 • Priority: P0 • Sources: FIG-LIB;VEL-11 |
| FIG-DS-02 — Component instances | Instances inherit from components while allowing controlled overrides and can be inspected for detachment or drift. | Detect rendered/source instances, props, variants, overrides, copied markup, and divergence; propose migrations back to the system. | Build component usage graph and structural similarity analysis.<br>Phase: P6 • Priority: P1 • Sources: FIG-LIB;FIG-CHECK |
| FIG-DS-03 — Variants and component properties | Variants, booleans, text, instance swaps, and exposed properties represent supported component states. | Map to typed props, variants, state machines, stories, and test matrices; agents change the supported API rather than patching individual screenshots. | Extract TypeScript/PropTypes/schema metadata and generate variant previews/tests.<br>Phase: P6 • Priority: P0 • Sources: FIG-LIB;FIG-DEV |
| FIG-DS-04 — Styles | Shared color, text, effect, and layout styles enforce consistency and allow centralized updates. | Translate to CSS variables, theme tokens, utility config, typography scales, component recipes, and platform style resources. | Build token/style adapters and usage graph.<br>Phase: P5 • Priority: P0 • Sources: FIG-LIB;VEL-10 |
| FIG-DS-05 — Variables | Variables store reusable values such as color, number, string, and boolean and can bind to properties. | Use a normalized design-variable graph sourced from code/config, not a separate design-only database. Edits produce source changes and validation. | Create DesignTokenService with adapters for CSS, Tailwind, Style Dictionary, theme objects, native platforms, and Figma imports.<br>Phase: P6 • Priority: P0 • Sources: FIG-VARS;VEL-10 |
| FIG-DS-06 — Variable collections | Variables are organized into collections with shared purpose and modes. | Represent collections such as color, spacing, typography, elevation, motion, breakpoints, density, brand, and platform; preserve aliases and provenance. | Define token manifest schema and collection adapters.<br>Phase: P6 • Priority: P1 • Sources: FIG-VARS |
| FIG-DS-07 — Modes | Variable modes express light/dark, brand, locale, density, device, platform, or other contextual value sets. | Render and verify all relevant modes across routes and components; changes must show affected combinations and fallback behavior. | Build mode matrix, preview controls, and automated capture.<br>Phase: P6 • Priority: P0 • Sources: FIG-VARS |
| FIG-DS-08 — Aliases | Variables can reference other variables to create semantic layers over primitive values. | Encourage primitive-to-semantic-to-component token layers and detect cycles, broken aliases, and raw-value bypasses. | Implement token dependency graph and lint rules.<br>Phase: P6 • Priority: P0 • Sources: FIG-VARS |
| FIG-DS-09 — Code syntax for variables | Figma variables can carry platform-specific code syntax names for developer handoff. | Maintain mapping from design semantics to actual code identifiers and files, supporting multiple platforms and migration aliases. | Store code bindings in token manifest and source index.<br>Phase: P6 • Priority: P1 • Sources: FIG-VARS;FIG-DEV |
| FIG-DS-10 — Libraries | Teams publish reusable components, styles, and variables so other files consume governed versions. | Treat libraries as versioned code packages/design-system repositories with release notes, compatibility, ownership, and dependency updates. | Integrate package managers, repository graph, semantic versions, and design previews.<br>Phase: P6 • Priority: P0 • Sources: FIG-LIB |
| FIG-DS-11 — Library publication | Changes to shared assets are reviewed and published for downstream use. | A design-system checkpoint produces package/version candidates, usage impact, visual matrix, migration guidance, and approval before release. | Build library release workflow and package/deployment adapters.<br>Phase: P7 • Priority: P1 • Sources: FIG-LIB |
| FIG-DS-12 — Library updates | Consumers see available updates and can review/accept them selectively. | Velocity should detect package/token/component updates, preview impact in candidate worktrees, and let teams adopt via generated migration checkpoints. | Create dependency update agent, compatibility report, and selective rollout controls.<br>Phase: P7 • Priority: P1 • Sources: FIG-UPD |
| FIG-DS-13 — Swap libraries | Design files can migrate instances/styles from one library to another. | Support source-level component/package replacement using codemods, prop mapping, token mapping, and visual/test verification. | Build migration maps and semantic codemod pipeline.<br>Phase: P7 • Priority: P1 • Sources: FIG-LIB |
| FIG-DS-14 — Assets panel | Figma exposes reusable components, local assets, and library resources for discovery. | Velocity's Design Lens should list source components, routes, icons, media, tokens, patterns, stories, and packages with usage and ownership. | Build semantic asset explorer over graph/index services.<br>Phase: P6 • Priority: P1 • Sources: FIG-DESIGN;VEL-11 |
| FIG-DS-15 — Dev Mode inspection | Developers inspect dimensions, layout, typography, colors, variables, assets, and code-relevant details. | Velocity combines browser-computed state with source definitions and framework semantics, showing what is rendered, where it comes from, and how to change it safely. | Create Inspect Lens with DOM/CDP, source maps, component graph, token graph, and code links.<br>Phase: P5 • Priority: P0 • Sources: FIG-DEV |
| FIG-DS-16 — Code snippets | Dev Mode generates framework/platform code snippets for selected designs. | Generate explanatory snippets or suggested usage from actual component APIs and source, clearly labeling generated examples versus canonical code. | Build snippet providers tied to component schemas and repository conventions.<br>Phase: P7 • Priority: P2 • Sources: FIG-DEV |
| FIG-DS-17 — Code Connect | Figma Code Connect maps design components to their real code implementations and usage examples. | Make component-to-rendered-element mapping a core Velocity service, enabling source edits, documentation, tests, and browser inspection from one semantic ID. | Implement framework adapters, source annotations, runtime IDs, and mapping diagnostics.<br>Phase: P6 • Priority: P0 • Sources: FIG-DEV |
| FIG-DS-18 — Dev resources | Design nodes can link to tickets, documentation, repositories, stories, or other development resources. | Allow every semantic artifact to attach issues, docs, API specs, database schemas, tests, owners, dashboards, and prior decisions. | Build generic ResourceLink service and provider integrations.<br>Phase: P6 • Priority: P1 • Sources: FIG-DEV |
| FIG-DS-19 — Annotations | Designers add structured annotations for behavior, accessibility, content, or implementation intent. | Use annotations as mission criteria and semantic constraints that agents must preserve or verify, not passive sticky notes. | Define typed annotation schemas and validation hooks.<br>Phase: P6 • Priority: P0 • Sources: FIG-DEV |
| FIG-DS-20 — Ready-for-dev sections | Frames/sections can be marked ready and filtered for developer focus. | Velocity derives readiness from plan completeness and required evidence, while allowing an authorized human to override with an explicit reason. | Implement readiness policy and status view.<br>Phase: P5 • Priority: P0 • Sources: FIG-DEV |
| FIG-DS-21 — Change comparison | Dev Mode can highlight design changes since a prior version or readiness point. | Compare Stable/Candidate at source, component, token, DOM, screenshot, accessibility, API, schema, and test levels. | Build multi-domain change index and compare editor.<br>Phase: P6 • Priority: P0 • Sources: FIG-DEV;FIG-REVIEW |
| FIG-DS-22 — Component playground | Developers can explore component properties and states interactively. | Generate live source-backed component playgrounds from stories/tests/prop schemas, with edits expressed as temporary props until a mission changes source. | Build custom editor or browser route for component matrices.<br>Phase: P7 • Priority: P1 • Sources: FIG-DEV |
| FIG-DS-23 — Focus view | Dev Mode can reduce design noise and focus on implementation-ready content. | Offer a Review/Inspect profile that hides authoring controls and centers source-linked behavior, specs, and evidence. | Create layout profile and context-driven views.<br>Phase: P6 • Priority: P2 • Sources: FIG-DEV |
| FIG-DS-24 — Design linting | Figma Check designs finds hard-coded values, wrong library assignments, detached components, contrast issues, and inconsistent spacing or radius. | Implement source-aware design checks for raw values, token misuse, detached/copied components, inaccessible contrast, inconsistent spacing, missing states, and responsive regressions. | Create DesignLintService that publishes Problems and autofix previews.<br>Phase: P6 • Priority: P0 • Sources: FIG-CHECK |
| FIG-DS-25 — Batch design fixes | Check designs can apply one, several, or all suggested fixes. | Agents may propose batch fixes grouped by semantic rule and impact; each batch gets source diff, visual matrix, and rollback rather than silent global mutation. | Use code actions/codemods and checkpoint workflow.<br>Phase: P6 • Priority: P0 • Sources: FIG-CHECK |
| FIG-DS-26 — Design analytics | Library analytics and API data can reveal component usage, adoption, detachments, and update status. | Measure design-system adoption and drift from source usage, package versions, token references, and runtime/rendered evidence. | Build privacy-aware design-system analytics and dashboards.<br>Phase: P8 • Priority: P2 • Sources: FIG-API |
| FIG-DS-27 — REST API | Figma's API exposes files, nodes, comments, versions, projects, components, variables, webhooks, and activity logs. | Provide a Velocity API for projects, missions, coworkers, work orders, artifacts, evidence, checkpoints, comments, decisions, graph entities, and automation events. | Design versioned local/cloud REST or RPC API with granular scopes.<br>Phase: P7 • Priority: P0 • Sources: FIG-API |
| FIG-DS-28 — Plugin API | Figma plugins inspect and mutate document graphs and extend workflows inside the product. | Velocity plugins should operate on semantic project resources and request source-level mutations through tools/checkpoints, not directly rewrite hidden UI state. | Expose extension/SDK APIs with capability grants and transactional edits.<br>Phase: P8 • Priority: P1 • Sources: FIG-PLUGIN |
| FIG-DS-29 — Agent-first design acceptance gate | The design layer is complete when a user can select semantic scope, state intent, preview source-backed alternatives, verify across modes/viewports/routes, review code and behavior evidence, and accept or roll back—without manually editing every layer. | This is the direct answer to the user's requirement: Figma-grade collaboration and systems thinking, but agents perform the actual cross-project edits. | Create end-to-end design transformation suites and source-of-truth assertions.<br>Phase: P6 • Priority: P0 • Sources: VEL-02;CUR-BROW;FIG-DEV |
