# 23. Figma Multiplayer Presence, Comments, and Shared Control

> Part of the [Velocity Platform Research & Integration Roadmap](README.md). Research snapshot: **15 July 2026**.

Figma's multiplayer model supplies the best patterns for presence, following, durable pinned discussion, lightweight live communication, roles, and reconnect behavior. Velocity applies these patterns to semantic software artifacts.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| FIG-COLLAB-01 — Real-time multiplayer presence | Figma shows active collaborators, identities, cursors, selections, and activity in the same file. | Show humans and coworkers spatially on the active artifact, code, system, data, or test Lens, with density controls and non-spatial list equivalents. | Implement presence service over project/session IDs, heartbeat, viewport, selection, and semantic resource references.<br>Phase: P5 • Priority: P0 • Sources: FIG-DESIGN;VEL-02 |
| FIG-COLLAB-02 — Live cursors | Collaborator cursors move in real time and use compact identity labels/colors. | Use cursors for humans; use restrained semantic markers for coworkers so automation does not create distracting pseudo-mouse theater. | Transmit human pointer state; derive coworker markers from active tool/resource events rather than synthetic motion.<br>Phase: P5 • Priority: P1 • Sources: FIG-DESIGN;VEL-02 |
| FIG-COLLAB-03 — Selection presence | Collaborators can see which frame, layer, or object another person has selected. | Show selected files, symbols, browser elements, routes, tests, checkpoints, or evidence objects as semantic selections. | Represent selection with ResourceRef plus optional spatial bounds and revision.<br>Phase: P5 • Priority: P0 • Sources: FIG-DESIGN |
| FIG-COLLAB-04 — Viewport presence | Presence can communicate what area/page/view a collaborator is looking at. | Track Lens, editor group, resource, browser route, zoom/scroll region, and Stable/Candidate context under privacy settings. | Publish coarse view state and avoid broadcasting sensitive text selections by default.<br>Phase: P5 • Priority: P1 • Sources: FIG-SPOT |
| FIG-COLLAB-05 — Spotlight/follow presenter | A user can follow another collaborator's viewport, page, zoom, and navigation until they stop following. | Velocity Follow Mode should follow a coworker's actual active resource and tool events, or a human's view, without stealing control unless explicitly synchronized. | Implement follow subscription, view translation, pause/peek controls, and independent exit.<br>Phase: P5 • Priority: P0 • Sources: FIG-SPOT;VEL-02 |
| FIG-COLLAB-06 — Follow coworker | Velocity's target behavior lets a user observe a named coworker's current work, checkpoint, tools, and history. | Combine spatial following with a structured side panel containing objective, current step, scope, terminal/browser/test activity, latest evidence, and blockers. | Build on presence and orchestration events; do not expose hidden chain-of-thought.<br>Phase: P5 • Priority: P0 • Sources: VEL-01;VEL-02 |
| FIG-COLLAB-07 — Presenter handoff | Spotlight can move attention between collaborators while preserving shared context. | Allow a human to hand presentation control to another human or coworker evidence replay, with clear current leader and one-click stop. | Create presentation session with leader, followers, permissions, and history.<br>Phase: P6 • Priority: P2 • Sources: FIG-SPOT |
| FIG-COLLAB-08 — Pinned comments | Comments attach to specific coordinates/objects and remain as durable discussion/work markers. | Velocity pins attach to semantic artifact identities with coordinates only as fallback; they can become work orders, decisions, review notes, or resolved annotations. | Store locator/source/route/revision/author/intent/status and render across relevant lenses.<br>Phase: P5 • Priority: P0 • Sources: FIG-COMMENTS;VEL-01 |
| FIG-COLLAB-09 — Comment threads | Comments support replies, mentions, timestamps, resolution, editing, deletion, reactions, and notifications. | Implement threads as audit-visible work context, with agent replies limited to commitments, evidence, and questions—not fabricated conversational filler. | Create CommentThread service and notification routing.<br>Phase: P5 • Priority: P0 • Sources: FIG-COMMENTS |
| FIG-COLLAB-10 — Mentions | Collaborators can mention people to request attention or assign follow-up. | Support mentions of humans, coworkers, teams, missions, and semantic artifacts; mentioning a coworker may propose assignment but policy/scope still applies. | Resolve identities and dispatch inbox events or assignment requests.<br>Phase: P5 • Priority: P1 • Sources: FIG-COMMENTS |
| FIG-COLLAB-11 — Reactions | Emoji reactions provide lightweight acknowledgement without adding replies. | Use compact acknowledgement states such as seen, approved, needs review, or blocked; keep optional emoji as secondary social metadata. | Implement reaction events without allowing them to replace formal approvals.<br>Phase: P6 • Priority: P3 • Sources: FIG-COMMENTS |
| FIG-COLLAB-12 — Resolve and reopen | Threads can be resolved when addressed and reopened when the issue returns. | A comment resolves only when linked work is accepted or the author explicitly dismisses it; new regressions can reopen with evidence. | Link thread state to work-order/checkpoint events and maintain audit history.<br>Phase: P5 • Priority: P0 • Sources: FIG-COMMENTS |
| FIG-COLLAB-13 — Comment filters | Comments can be filtered by status, participant, page, and related context. | Filter Velocity annotations by unresolved, assigned coworker, mission, Lens, route, risk, checkpoint, author, and age. | Build indexed query model and saved filters.<br>Phase: P6 • Priority: P2 • Sources: FIG-COMMENTS |
| FIG-COLLAB-14 — Comment notifications | Authors and mentioned collaborators receive notifications about replies, mentions, resolution, or changes. | Route only actionable events to the attention inbox; offer digest, mute, watcher, and organization policy controls. | Use notification service and per-thread subscriptions.<br>Phase: P5 • Priority: P1 • Sources: FIG-COMMENTS |
| FIG-COLLAB-15 — Ephemeral cursor chat | Figma cursor chat provides temporary text attached to a user's cursor during live collaboration. | Velocity may offer ephemeral 'point and say' messages for humans during live review, but not as the primary work model and not for agent control. | Implement opt-in transient presence messages with no durable project semantics unless converted to a pin.<br>Phase: P7 • Priority: P3 • Sources: FIG-CURSOR |
| FIG-COLLAB-16 — Audio/live collaboration integration | Collaborative design workflows may use external or integrated voice/video alongside shared context. | Velocity should integrate links and shared presentation state rather than becoming a meeting platform; decisions and work still become durable artifacts. | Provide meeting-context hooks and decision capture APIs.<br>Phase: P8 • Priority: P3 • Sources: FIG-DESIGN |
| FIG-COLLAB-17 — Permissions and roles | Files/teams support roles such as owner, editor, viewer, and commenter, with organization-level controls. | Define Owner, Maintainer, Developer, Reviewer, Commenter, Viewer, Operator, and Auditor permissions plus agent authority profiles. | Build RBAC/ABAC over projects, tools, environments, artifacts, and protected actions.<br>Phase: P6 • Priority: P0 • Sources: FIG-HANDOFF |
| FIG-COLLAB-18 — View-only collaboration | Viewers can inspect and comment without editing source. | Provide browser/web review clients where stakeholders can inspect Stable/Candidate, replay evidence, comment, and decide within role, without local repository access. | Build read-only project gateway and signed artifact delivery.<br>Phase: P7 • Priority: P1 • Sources: FIG-HANDOFF |
| FIG-COLLAB-19 — Presence privacy | Collaboration systems must decide what location, selection, activity, and identity information is shared. | Offer organization/project/user settings for cursor precision, file names, terminal/browser visibility, and presence retention; sensitive environments may show only coarse status. | Implement presence redaction and policy evaluation.<br>Phase: P6 • Priority: P0 • Sources: FIG-DESIGN |
| FIG-COLLAB-20 — Offline and reconnect behavior | Clients may lose connection and later resume, requiring state reconciliation and clear presence expiry. | Expire stale presence quickly, queue durable comments/decisions locally, reconcile CRDT text where enabled, and never imply an offline coworker is actively working. | Use presence leases, offline queue, conflict rules, and reconnect snapshots.<br>Phase: P6 • Priority: P1 • Sources: FIG-DESIGN |
| FIG-COLLAB-21 — Collaboration acceptance gate | Real collaboration requires identity, roles, presence, follow, durable comments, notifications, offline recovery, audit, and conflict-safe shared state. | The current invite demo and same-window document sharing do not meet this gate; network collaboration must be tested under latency and reconnect. | Adopt Yjs or equivalent selectively, a collaboration server, presence protocol, and end-to-end fault tests.<br>Phase: P6 • Priority: P0 • Sources: VEL-09;VEL-12;FIG-DESIGN |

# 24. Figma Branching, Review, Version History, and Handoff

Figma's branch/review model closely matches Velocity's Stable/Candidate concept. The implementation, however, must be backed by Git worktrees, immutable evidence, typed approvals, source/visual comparison, and safe integration.

| Capability | How it works | Velocity translation | Implementation / sequencing |
| --- | --- | --- | --- |
| FIG-REV-01 — Branch from main | Figma branches create isolated copies of a file where changes can proceed without affecting the main file. | Velocity candidates should be Git branches/worktrees, browser profiles, terminals, and environment state isolated from Stable. | Use WorktreeService and CandidateEnvironment with explicit base revision.<br>Phase: P5 • Priority: P0 • Sources: FIG-BRANCH;CUR-WT |
| FIG-REV-02 — Named branch purpose | A branch carries a name, description, author, status, and relation to main. | Expose human-facing mission/coworker/checkpoint identity while recording implementation branch/worktree metadata underneath. | Store BranchBinding on work orders and checkpoints.<br>Phase: P5 • Priority: P1 • Sources: FIG-BRANCH;VEL-02 |
| FIG-REV-03 — Update from main | Branches can pull changes from the main file and reconcile conflicts before review or merge. | Provide controlled candidate refresh, base-drift warning, semantic conflict preview, and mandatory re-verification after integration. | Implement update/rebase workflow and invalidation rules.<br>Phase: P5 • Priority: P0 • Sources: FIG-BRANCH;VS-SCM |
| FIG-REV-04 — Branch checkpoints | Figma preserves checkpoints/history within a branch so progress can be recovered. | Create intermediate Git commits and evidence snapshots at meaningful safe points, separate from review-ready checkpoints. | Add internal recovery checkpoints with retention and promotion to formal checkpoint.<br>Phase: P5 • Priority: P1 • Sources: FIG-BRANCH;VEL-02 |
| FIG-REV-05 — Request review | A branch author can ask collaborators to review proposed changes. | Coworkers request checkpoint review only after required evidence passes or explicitly mark missing/failed gates; the inbox targets qualified reviewers. | Create ReviewRequest with scope, reviewers, readiness, evidence, and deadline.<br>Phase: P5 • Priority: P0 • Sources: FIG-REVIEW;VEL-02 |
| FIG-REV-06 — Reviewer assignment | Specific people or teams can be invited to review, comment, approve, or request changes. | Select reviewers by ownership, role, risk, protected areas, and availability; agents may review only within declared authority. | Build review policy resolver and inbox routing.<br>Phase: P6 • Priority: P0 • Sources: FIG-REVIEW |
| FIG-REV-07 — Side-by-side comparison | Reviewers compare branch and main versions visually and inspect changed areas. | Use synchronized Stable/Candidate browser views, native source diffs, system/data/test deltas, and evidence timeline. | Build CompareEditor orchestrating multiple native/custom editors.<br>Phase: P5 • Priority: P0 • Sources: FIG-REVIEW;VEL-02 |
| FIG-REV-08 — Overlay comparison | Visual branches can be overlaid to reveal movement or visual differences. | Offer opacity slider, blink, pixel/perceptual diff, DOM box changes, and route/viewport synchronization for UI checkpoints. | Build visual comparison renderer over deterministic captures.<br>Phase: P5 • Priority: P1 • Sources: FIG-REVIEW |
| FIG-REV-09 — Change navigation | Review surfaces guide users through changed objects or areas rather than forcing a full manual scan. | Order source and visual changes by semantic scope, risk, user-facing impact, failing evidence, and dependency. | Create ChangeIndex from Git diff, semantic graph, browser diff, and diagnostics.<br>Phase: P5 • Priority: P0 • Sources: FIG-REVIEW |
| FIG-REV-10 — Review comments | Reviewers pin comments to proposed changes and discuss revisions before merge. | Anchor notes to candidate revision, semantic resource, diff range, browser element, test result, or evidence artifact, and preserve across revisions where identity remains valid. | Use versioned AnnotationTarget and reanchor service.<br>Phase: P5 • Priority: P0 • Sources: FIG-REVIEW;FIG-COMMENTS |
| FIG-REV-11 — Request changes | A reviewer can send a branch back with explicit feedback instead of rejecting the whole effort. | Translate notes into revision directives attached to the existing work order; the coworker resumes from checkpoint with changed criteria and review anchors. | Implement revise transition and directive queue.<br>Phase: P5 • Priority: P0 • Sources: FIG-REVIEW;VEL-12 |
| FIG-REV-12 — Approve | Authorized reviewers can approve when the branch meets expectations. | Separate code review, product approval, security approval, data approval, and deployment approval; required approvals derive from policy/risk. | Create typed approvals with signer, scope, revision, policy, expiration, and invalidation.<br>Phase: P6 • Priority: P0 • Sources: FIG-REVIEW |
| FIG-REV-13 — Merge | Approved branch changes merge into main and update the shared file. | Acceptance advances Velocity Stable only after source merge, combined verification, manifest finalization, and optional deployment gates; partial failure triggers compensation. | Implement acceptance transaction and merge queue.<br>Phase: P6 • Priority: P0 • Sources: FIG-BRANCH;VEL-02 |
| FIG-REV-14 — Conflict resolution | Conflicts between branch and main are surfaced for resolution, often with object-level context. | Combine Git merge conflicts with semantic ownership, visual deltas, schema conflicts, and design-token collisions; automation proposes but does not conceal ambiguity. | Use merge editor, conflict analyzer, and Decision Sheets.<br>Phase: P6 • Priority: P0 • Sources: FIG-BRANCH;VS-SCM |
| FIG-REV-15 — Archive and restore branches | Branches can be archived and later restored instead of immediately deleted. | Archive completed, rejected, superseded, or abandoned candidates with retention rules and restore capability; prune execution resources separately. | Store metadata and commits while deleting worktrees/processes/browser partitions safely.<br>Phase: P6 • Priority: P1 • Sources: FIG-BRANCH |
| FIG-REV-16 — Version history | Figma records named versions and lets users inspect, duplicate, or restore earlier file states. | Velocity's timeline spans Git commits, checkpoints, evidence, decisions, deployments, data migrations, and rollbacks; restore semantics depend on artifact type. | Build project history service and recovery planner.<br>Phase: P6 • Priority: P0 • Sources: FIG-DESIGN;VEL-02 |
| FIG-REV-17 — Autosave history | Frequent background versions protect work without requiring deliberate saves. | Use native working-copy backups for human edits and tool-boundary commits/state snapshots for agents; do not create noisy formal checkpoints for every token. | Coordinate backups, internal snapshots, and retention.<br>Phase: P3 • Priority: P1 • Sources: VS-EDITOR;FIG-DESIGN |
| FIG-REV-18 — Ready-for-development status | Figma can mark sections or frames ready for development and notify developers. | Velocity should mark missions/checkpoints as Planned, Ready to Build, In Progress, Ready to Review, Approved, Ready to Ship, Live, or Rolled Back based on evidence—not manual labels alone. | Implement status state machine with policy-derived transitions.<br>Phase: P5 • Priority: P0 • Sources: FIG-DEV |
| FIG-REV-19 — Developer handoff | Developers inspect measurements, styles, variables, assets, code references, annotations, and change status without editing the design. | Velocity handoff is continuous: source, running behavior, design tokens, API/schema data, tests, evidence, and ownership are linked in one project graph. | Build semantic handoff views and deep links among browser, code, graph, tests, and checkpoints.<br>Phase: P6 • Priority: P0 • Sources: FIG-HANDOFF;VEL-02 |
| FIG-REV-20 — Review-state invalidation | When a reviewed branch changes, prior approval or reviewed regions may no longer be valid. | Invalidate only affected file/hunk/evidence approvals when possible; invalidate whole-checkpoint approvals for base changes, policy changes, or critical dependencies. | Implement dependency-aware review stamps tied to immutable revisions.<br>Phase: P6 • Priority: P0 • Sources: FIG-REVIEW;VS-AGENTSW |
| FIG-REV-21 — Branch analytics | Organizations may need metrics on branch age, review time, rework, conflicts, and adoption. | Measure work-order cycle time, review latency, evidence failure, merge conflict, rollback, and human intervention without using activity as performance surveillance. | Build privacy-aware product/project analytics with clear definitions.<br>Phase: P8 • Priority: P2 • Sources: FIG-API |
| FIG-REV-22 — Branch/review acceptance gate | A candidate workflow is complete only when isolation, update-from-main, conflict handling, review anchors, typed approvals, merge, archive, restore, and review invalidation work across source and visual evidence. | Do not implement 'Stable/Candidate' as labels over shared mutable state; it must be backed by immutable revisions and isolated execution. | Create end-to-end branch/review test matrix.<br>Phase: P5 • Priority: P0 • Sources: FIG-BRANCH;FIG-REVIEW;VEL-02 |
